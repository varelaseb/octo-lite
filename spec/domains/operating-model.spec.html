octo-lite operating model

Specifications / operating model

Authority, shared invariants, canonical navigation, and signed decisions.

Visual and prose authority

A diagram normatively owns the states and relationships it explicitly encodes.

Prose normatively owns exact constraints, exceptions, and behavior a diagram cannot express precisely.

Each rule has one primary representation; the other may clarify or link but should not restate it.

Any mismatch invalidates the suite and blocks shaping until reconciled; neither representation silently wins.

Source authority and gate flow

Intended behavior

octo-lite makes delivery trustworthy and observable without becoming a service or a second operational state machine.

The optimization order is trustworthy delivery, low operator latency, throughput, then token and compute cost.

Linear owns high-level issue identity, decisions, context, non-goals, dependencies, and lifecycle state.

Canonical specs and ADRs own detailed behavior, acceptance rules, interfaces, edge cases, and constraints.

One evolving draft pull request carries shaping and implementation without creating divergent branches of truth.

Every behavior-changing implementation follows spec-driven red then green unless shaping records a narrow operator-approved waiver.

Refactoring is not part of the implementation loop; it arrives through reviewer-flagged refactor-only fix passes verified green-stays-green per the delivery lifecycle.

Domain concepts

Operator
Fable is the persistent meta-operator and sole writer of the reconciled whole-operation view.
Stream orchestrator
One persistent Claude Opus session owns exactly one issue or one epic coordination layer and reports to exactly one parent.
Epic orchestrator
For a multi-issue epic, one dedicated Opus parents the issue orchestrators and owns cross-issue dependencies, topology, carrier integration, and epic outcome.
Issue orchestrator
Every implementation issue has exactly one dedicated Opus that owns its issue scope, branch, gates, workers, and status.
Worker pass
One fresh disposable instance performs one implementation, fix, review, re-review, QA capture, or QA review pass.
Brief
A parent-owned Markdown artifact containing hard invariants, owned scope, standing authority, and must-escalate conditions.
Status
A child-owned Markdown artifact containing current gate, health, exact issue and PR facts, material change, and next action.
Receipt
A compact machine-readable fact binding a session or gate to exact role, model, instructions, repo, issue, spec, PR, and HEAD inputs.
Gate
A durable exact-revision artifact required before a downstream workflow transition.

Optional tooling boundary

Runtime boundary

octo-lite is optional installed developer tooling, not a product dependency or service.

It owns no daemon, API, database, scheduler, queue, mirror, poller, cache service, or unowned background runtime.

Small helpers execute inside explicit operator, orchestrator, or workflow sessions.

A target product must build, test, deploy, and remain understandable if octo-lite, spec-chat, and the optional dashboard are removed.

Instruction ownership

Instruction ownership

octo-lite owns generic lifecycle, roles, gates, TDD, review mechanics, model policy, and reusable skills.

A target repo root AGENTS.md exclusively owns repo-specific commands, architecture, validation, deployment, fixtures, and conventions.

A target root AGENTS.md contains only that target's instructions and is its sole repo instruction source.

A target root CLAUDE.md is a tracked relative symlink to root AGENTS.md when Claude tooling needs that entry point.

Target instructions remain versioned inside each worktree while generic octo-lite skills, roles, and workflows install by symlink from one canonical source.

Claude and Codex memory is non-authoritative and may contain only dated evidence-backed gotchas plus durable source links.

A memory entry contains only date, scope, observed failure, and durable source or fix link.

Any role may write a qualifying gotcha; the bounded stream orchestrator prunes or promotes entries when the stream closes.

Every LLM-facing role, skill, launch instruction, brief, handoff, verdict, receipt, and status artifact is extremely concise and may sacrifice grammar for concision.

LLM-facing instructions and generated text use no em dash or en dash characters.

Generated artifacts reference pinned canonical sources and exact revisions rather than restating specs, issues, roles, logs, or earlier artifacts.

Each artifact carries only fields needed for its authority, gate, recovery, or next consumer, but never omits a correctness-critical fact.

Detailed prose appears only for a finding, failure, ambiguity, operator decision, or explicit request.

No arbitrary word, character, or token cap may truncate required facts; compact schemas and source references enforce leanness.

Artifact format ownership

Artifact formats

octo-lite and participating target repos use spec-chat HTML for canonical human-reviewed domain behavior and ADRs.

Markdown remains canonical for agent-native instructions and lifecycle-bound human context.

TOML is the default for octo-lite-owned machine configuration and current control state.

JSONL is the append-only workflow telemetry format.

JSON or another native format remains only at an external tool, API, schema, or target-consumer boundary.

Screenshots and necessary video remain native media, and generated HTML views remain non-authoritative projections.

Machine state and events carry schema versions, TOML updates atomically, and JSONL appends under a small lock.

Format never determines authority.

Constraints

Prompts, roles, skills, and receipts remain extremely concise and use progressive disclosure.

Skills use no em dash or en dash characters.

Generic octo-lite mechanics are not copied into target repos.

TopicFinder-specific paths, credentials, fixtures, states, and dashboard mechanics do not live in generic octo-lite prompts or skills.

The optional hosted dashboard is read-only and never becomes a second control plane.

Source systems and control artifacts remain authoritative if every generated view is unavailable.

A cross-cutting ruling propagates by a full semantic sweep of every durable surface, reading each file whole and judging its meaning under Decision 108 discipline with one writer per file, never a grep-only or pattern-only pass.

Non-goals

Building an octo-lite service, scheduler, event bus, tracker mirror, orchestration database, or watcher fleet is out of scope.

Building a policy compiler, role registry service, giant spawn manifest, or inheritance engine is out of scope.

Running Langfuse or OpenTelemetry infrastructure is out of scope.

Building a broad Cartesian workflow conformance matrix is out of scope.

Making transcript replay, memory, pane pixels, or queued text a correctness dependency is out of scope.

Building an agent-callable dead-owner recovery command, liveness-probe authority, or cryptographic operator-authentication service is out of scope.

Depending on Codex custom-agent discovery for role identity or fallback is out of scope.
