octo-lite operating model

Specifications / operating model

Canonical workflow behavior for lightweight, trustworthy, spec-driven delivery.

Intended behavior

octo-lite makes delivery trustworthy and observable without becoming a service or a second operational state machine.

The optimization order is trustworthy delivery, low operator latency, throughput, then token and compute cost.

Linear owns high-level issue identity, decisions, context, non-goals, dependencies, and lifecycle state.

Canonical specs and ADRs own detailed behavior, acceptance rules, interfaces, edge cases, and constraints.

One evolving draft pull request carries shaping and implementation without creating divergent branches of truth.

Every behavior-changing implementation follows spec-driven red, green, refactor unless shaping records a narrow operator-approved waiver.

Domain concepts

Operator
Fable is the persistent meta-operator and sole writer of the reconciled whole-operation view.
Stream orchestrator
One persistent Claude Opus session owns exactly one issue or one epic coordination layer and reports to exactly one parent.
Epic orchestrator
For a multi-issue epic, one dedicated Opus parents the issue orchestrators and owns cross-issue dependencies, topology, carrier integration, and epic outcome.
Issue orchestrator
Every implementation issue has exactly one dedicated Opus that owns its issue scope, branch, gates, workers, and status.
Worker pass
One fresh disposable instance performs one implementation, fix, review, re-review, QA capture, or QA review pass.
Brief
A parent-owned Markdown artifact containing hard invariants, owned scope, standing authority, and must-escalate conditions.
Status
A child-owned Markdown artifact containing current gate, health, exact issue and PR facts, material change, and next action.
Receipt
A compact machine-readable fact binding a session or gate to exact role, model, instructions, repo, issue, spec, PR, and HEAD inputs.
Gate
A durable exact-revision artifact required before a downstream workflow transition.

Rules and invariants

Runtime boundary

octo-lite is optional installed developer tooling, not a product dependency or service.

It owns no daemon, API, database, scheduler, queue, mirror, poller, cache service, or unowned background runtime.

Small helpers execute inside explicit operator, orchestrator, or workflow sessions.

A target product must build, test, deploy, and remain understandable if octo-lite, spec-chat, and the optional control panel are removed.

The optional control panel lives inside octo-lite under control-panel/, with frontend dependencies and build output isolated from the core workflow.

It is not a sibling repository and does not require independent schema-version coordination with octo-lite.

Core installation and workflow execution do not build, launch, or depend on the panel unless explicitly invoked.

Hosted mode remains a static read-only projection; local operator mode may edit only approved canonical octo-lite files, validate, and show a git diff.

Local edit paths are exactly roles/*.md, roles.toml, profile/AGENTS.md, and skills/*/SKILL.md.

Generated adapters, workflows, target-repo instructions, specs, and runtime state remain read-only in the panel.

A save validates and shows the git diff but never commits or pushes; no per-file edit registry exists.

The role inspector shows current roles, model, effort, mode, Linear mappings, actual resolved skills, workflow spawn graph, symlink health, editable canonical role prompts, and the generic system or profile prompt.

Validated prompt or mapping changes affect only newly spawned sessions; existing sessions remain pinned to their launch contract.

The panel derives target values from existing repo instructions, specs, git, Linear, PRs, and evidence manifests; a target has no control-panel adapter by default.

Only a shaped, proven non-derivable presentation mapping may add .octo-lite/control-panel.toml; it contains no agent instructions, lifecycle state, secrets, or product dependency.

Instruction ownership

octo-lite owns generic lifecycle, roles, gates, TDD, review mechanics, model policy, and reusable skills.

A target repo root AGENTS.md exclusively owns repo-specific commands, architecture, validation, deployment, fixtures, and conventions.

A target root AGENTS.md contains only that target's instructions and is its sole repo instruction source.

A target root CLAUDE.md is a tracked relative symlink to root AGENTS.md when Claude tooling needs that entry point.

Target instructions remain versioned inside each worktree while generic octo-lite skills, roles, and workflows install by symlink from one canonical source.

Claude and Codex memory is non-authoritative and may contain only dated evidence-backed gotchas plus durable source links.

A memory entry contains only date, scope, observed failure, and durable source or fix link.

Any role may write a qualifying gotcha; the bounded stream orchestrator prunes or promotes entries when the stream closes.

Every LLM-facing role, skill, launch instruction, brief, handoff, verdict, receipt, and status artifact is extremely concise and may sacrifice grammar for concision.

LLM-facing instructions and generated text use no em dash or en dash characters.

Generated artifacts reference pinned canonical sources and exact revisions rather than restating specs, issues, roles, logs, or earlier artifacts.

Each artifact carries only fields needed for its authority, gate, recovery, or next consumer, but never omits a correctness-critical fact.

Detailed prose appears only for a finding, failure, ambiguity, operator decision, or explicit request.

No arbitrary word, character, or token cap may truncate required facts; compact schemas and source references enforce leanness.

Role and model topology

The complete LLM role roster is meta-operator, orchestrator, shaping-reviewer, implementer, code-reviewer, qa-capture, qa-reviewer, and reconciler.

The issue orchestrator performs shaping mode by resolving the issue-shaper skill in its own persistent session, never a second persistent role or tab.

Every fix uses a fresh implementer instance and every re-review uses a fresh code-reviewer instance rather than separate role contracts.

Every nested stream owner uses the same orchestrator contract.

Every issue has one and only one dedicated Opus issue orchestrator.

A multi-issue epic adds a separate Opus epic orchestrator above its issue orchestrators, producing two Opus layers below Fable.

Fresh workers report to their issue orchestrator; issue orchestrators report to the epic orchestrator; the epic orchestrator reports upward to Fable or its declared parent.

A parent flattens relevant effective rules into the child brief; the child acknowledges them only at spawn or after a material brief revision.

Acceptance recording, merge, evidence publication, Linear transition, preproduction promotion, and live traffic shift are deterministic operator-authorized helpers rather than LLM roles.

The control panel displays deterministic gates and helpers separately from LLM roles.

The meta-operator uses exact claude-fable-5 at xhigh effort in auto permission mode.

Fable 5 uses its default 1M context without a redundant [1m] suffix.

An unavailable Fable 5 fails visibly and is never silently replaced by Opus or another model.

Every stream or subordinate orchestrator uses exact claude-opus-4-8[1m] at high effort in auto permission mode.

An unavailable orchestrator model fails visibly and is never silently replaced by another version or family.

Implementation and every fix pass use a fresh exact Anthropic Sonnet 5 instance at xhigh effort.

Code review and every re-review use a fresh GPT-5.6 Sol instance at high effort.

The final post-grill shaping review uses a fresh GPT-5.6 Sol instance at xhigh effort on the fast service tier.

QA capture uses a fresh exact Anthropic Sonnet 5 instance at high effort.

QA review uses a fresh GPT-5.6 Sol instance at high effort on the fast service tier.

Each reconciliation sweep uses a fresh exact claude-sonnet-5 instance at high effort with no fallback.

Deterministic helpers perform exact checks and provide one compact normalized read-only snapshot; Sonnet classifies only changed, missing, stale, contradictory, or needs_fable.

The reconcile gateway reads the exact snapshot file bytes from a regular contained path and hashes them, rejecting a missing, substituted, corrupt, or digest-mismatched snapshot before any worktree or provider launch.

Reconcile PR facts bind normalized PR state, review decision, and status check rollup outcomes alongside head, base, and branch, so a change in any of them changes the snapshot fingerprint.

Normalized status check facts sort on name, current status, and outcome so duplicate-name checks in any API order, or with differing outcomes or in-progress states, never collapse or reorder the same rollup's fingerprint.

The reconciler's canonical spec and ADR sweep requires every declared canonical path to exist as a readable regular file at the exact control HEAD, and a missing, renamed, unreadable, non-file, or empty resulting source set fails the sweep closed before any snapshot or worktree is produced.

Each target repo root AGENTS.md declares its own canonical spec and ADR paths as exact comma-separated Canonical spec paths and Canonical ADR paths lines, and octo-lite's own AGENTS.md declares its own paths the same way rather than a hardcoded constant.

A target initialization template declares exactly one Canonical spec paths placeholder and exactly one Canonical ADR paths signal in root AGENTS.md alongside the spec-format signal.

Shaping replaces the spec placeholder with exact target-owned canonical paths before readiness, and no role, skill, workflow, or octo-lite dependency is copied into the target.

The canonical sweep fails closed on an absent declaration, a missing, non-blob, empty, duplicate, malformed, escaped, or symlinked declared path, or a path declared under both the spec and ADR label.

Canonical ADR paths: none is the sole explicit no-ADR representation and binds an empty ADR blob list rather than a path.

Any other empty value, sentinel, a mix of none and a path, or a repeated Canonical ADR paths declaration fails the sweep closed, and at least one canonical spec path remains mandatory.

Zero or more than one signal line for either label fails the sweep closed, a prefix lookalike line never counts toward that total, and duplicate paths within one otherwise valid line still fail closed.

A sweep refetches and validates current Linear, GitHub PR, target HEAD, and canonical spec and ADR facts before any unchanged decision or snapshot persistence, and the compared fingerprint binds all of them rather than only declared stream facts.

The sweep captures one target HEAD and passes it as the gateway's expected HEAD, and the gateway compares its own freshly read current repo HEAD against that expected HEAD before validating any source or creating a worktree.

Any HEAD change between capture and gateway comparison, including an unrelated commit that leaves canonical blobs unchanged, fails before any provider launch and persists no snapshot, receipt, result, worktree, or sweep state.

The persisted receipt and the child worktree bind the same expected HEAD.

A snapshot, sweep state, and worktree are persisted only after every bound fact validates; a failed target HEAD read or canonical sweep leaves none of them behind.

The sweep never persists snapshot.md under the final sweep directory before Linear, PR, HEAD, spec, and ADR validation all succeed, and a failing gateway leaves no sweep artifact directory or state behind.

A temporary input required only for digest verification lives outside the final sweep directory and is removed on every failure, while a successful sweep still persists its atomic snapshot, receipt, result, and no-op state.

The persisted receipt binds reconcile.snapshot_path to the final persisted snapshot.md, never the temporary digest-verification input, and that exact file exists with matching bytes and digest before bootstrap starts.

Every caught pre-provider failure, including a receipt persistence failure that happens after snapshot persistence, leaves no final sweep snapshot, receipt, or sweep directory and no pending digest-verification input behind.

Reconciler bootstrap opens the exact file at the receipt-bound reconcile.snapshot_path, hashes its bytes, and refuses acknowledgment when that file is missing or its digest does not match reconcile.snapshot_digest.

Sonnet cannot mutate sources, override deterministic mismatches, investigate open-endedly, or silently resolve ambiguity.

Missing or unparseable input and semantic ambiguity return needs_fable; Fable judges the case.

Orchestrated Claude Workflows are the primary loop runtime rather than an adapter target.

Each roles/<role>.md is the sole canonical prose contract for that role.

roles.toml exclusively owns provider, exact model, effort, permission mode, tools, required and conditional skills, and contract path.

Role prose owns responsibilities, authority, stop conditions, inputs, and outputs without repeating machine configuration.

One small resolver combines the canonical role contract and machine mapping before every Workflow agent invocation or Sol relay.

Claude Markdown adapters are generated launcher inputs and diagnostic surfaces, never alternate role-bearing spawn paths.

Codex has no generated custom-agent adapter; Codex custom agents are not part of correctness or fallback for any role.

An OpenAI-backed role pass is one explicit codex exec relay launched by octo-launch, carrying exact model, effort, service tier, worktree, receipt, and the canonical contract as the exec prompt, and it never discovers or depends on a Codex custom agent.

The relay fails closed unless the exact effective provider, model, and effort are independently read back from the Codex session's own rollout record before any mutation.

Claude Workflows load canonical role prose verbatim and Sol relays receive the same prose without copied workflow literals.

The launcher loads every required skill plus only matched conditional skills and records the actual resolved set.

The launch receipt records role path and git blob, roles.toml revision, exact runtime values, and resolved skills.

Adapter generation fails if a target platform cannot express a required tool, mode, or contract field.

roles.toml contains no target rules, issue scope, live state, secrets, or operator decisions.

One path-filtered CI dry run proves role blob, map revision, model, mode, skills, target instruction blob, repo, worktree, and starting HEAD reached the spawned session.

If the Workflow runtime cannot read canonical files directly, a deterministic prebuilt Workflow injects the exact role text and source blob without handwritten prompt literals.

A worker instance is never resumed for another pass and --last is prohibited.

A completed orchestrator stream archives its control record and terminates rather than accepting later work.

Shaping contract

Shaping begins from the complete current Linear issue for Linear-tracked work.

The issue shaper grills one unresolved question at a time with a recommended answer and records each resolved behavior immediately.

Shaping is not complete without an aligned Linear issue, canonical spec, evolving draft PR, and final fresh shaping-review verdict.

The shaping reviewer reads canonical specs and ADRs, relevant conversation logs through a pinned cutoff, Linear, repo instructions, the same draft-PR readiness HEAD, and the TDD contract.

Linear Shaped is the canonical readiness state and is set only after a clear final shaping verdict.

A verified issue orchestrator in shaping mode may move Ideas or Todo to Shaped only after live readback of one canonical clear shaping verdict at the exact PR HEAD and a linear:<issue>:<fingerprint> binding for the exact current Linear issue.

Existing delivery and operator boundaries remain unchanged.

The final shaping verdict is a durable artifact on the evolving draft PR, not a separate readiness file.

The verdict binds the relevant Linear content fingerprint, PR and shaping HEAD, spec and ADR blobs, conversation-log references and cutoff, clear or blocking result, and reviewer receipt reference.

Linear links the PR verdict without duplicating its exact-revision proof.

Implementation launch refetches Linear, the PR, HEAD, specs, logs, and verdict; any mismatch returns the issue to shaping.

Shaping and code review each publish one concise machine-readable PR verdict comment through a deterministic helper.

The comment records review type, clear or blocking verdict, exact HEAD, bound inputs, findings, and reviewer receipt reference.

No formal GitHub Review API operation, reviewer-identity preflight, check run, or fallback tree is required for workflow correctness.

Documentary gaps return to the shaper, missing judgments return to the operator grill, and material reconciliation requires another fresh shaping review.

The implementer treats the spec as detailed behavioral law and Linear as high-level decision, context, identity, non-goal, and status law.

Conversation logs are shaping-review evidence, not implementation instructions.

If Linear high-level law conflicts with canonical spec or ADR law, neither silently wins: stop implementation and return to shaping until both and the evolving PR agree.

Prompt TDD

Every prompt, role, or skill instruction change starts with a behavior contract naming exact inputs, the required observable decisions, actions, or fields, forbidden behavior, and acceptable proof.

A fresh model session runs the smallest scenario against the actual unchanged instructions first; only a genuinely wrong or ambiguous decision is red.

A missing file, module, export, or script is not a valid prompt red.

A different fresh model session runs the same scenario against the changed instructions and judges the observable decision against the contract for green, then narrow regression scenarios run for adjacent behavior the change could shift.

Deterministic tests cover loading, wiring, schema, invariants, and forbidden characters; they judge structure, never prompt judgment quality.

An exact prose snapshot or keyword presence is never behavioral proof.

Red chronology that cannot be honestly reconstructed must be stated as such, never fabricated.

Eval evidence stays a lean one-off scenario record; no eval service or general harness.

Delivery loop

shape and reconcile
  -> fresh shaping review
  -> implement with spec-derived red and green proof
  -> fresh code review
  -> fresh fix when blocking
  -> fresh re-review
  -> QA capture when user-facing
  -> fresh QA review
  -> operator acceptance
  -> merge and lifecycle transition

Every fix changes HEAD and requires a fresh code review before QA.

At most three fresh review and fix cycles may run before the issue returns to shaping.

Behavior or contract changes invalidate review and QA while packet-only wording changes do not.

No agent may infer human acceptance, approve as the human, or merge without the operator-owned acceptance artifact for the same verdict and HEAD.

Branch and release topology

The same draft PR continues from shaping through implementation.

The shaping readiness HEAD remains immutable evidence while implementation advances the PR HEAD.

Sequential, stacked, or parallel issue delivery is chosen by the accountable epic orchestrator from actual dependencies, collision risk, review cost, and throughput, never by a fixed programmatic default.

The train manifest records the current chosen topology and revision; it does not select or enforce topology.

Member PRs integrate into the declared carrier or predecessor, never directly into main; the carrier is the train's sole final-main PR.

After member integration, the orchestrator proves reviewed-HEAD ancestry and effective diff before preserving downstream gates.

The stream orchestrator has standing authority to select and revise main, stacked-branch, or integration-train topology without per-change approval.

Topology changes escalate only when ship grouping, product scope, or material risk changes.

Authorized PR retargeting preserves stacking and release trains but must record topology revision, merge base, ancestry, effective diff, and invalidated downstream gates.

A multi-PR train has one carrier PR and one machine-readable manifest listing member Linear issues, PRs, pinned HEADs, order, topology revision, integration branch, target, owner, and gate state.

The train manifest owns topology only and never duplicates product scope or lifecycle truth.

Linear lifecycle

Gate-critical reads use one exact Linear helper returning the complete issue, stable identity, revision, and current state.

A mechanical transition compares expected state, mutates once, reads back, updates stream status, and notifies the parent before reporting success.

A partial transition failure retries only unfinished idempotent steps and never blindly repeats a completed state mutation.

Stream orchestrators may perform only Todo -> In Progress -> Awaiting Accept through that helper.

Acceptance, merge, staging, preproduction, and live transitions remain operator-owned.

Acceptance inside a stack or release train remains an acceptance-ledger fact while Linear stays Awaiting Accept until the commit reaches main.

Generic octo-lite defines only that post-acceptance deployment and promotion states are durable, operator-authorized, and evidence-backed; the exact state names and the concrete facts that trigger each one are target-owned Linear state mapping, defined in the target repo, never in generic octo-lite.

User-story surface

The user-story catalog is a product-surface artifact and a QA verification contract for full-pipeline checks.

The story record is not QA evidence and does not embed screenshots, videos, verdicts, proof links, or acceptance state.

A participating target owns one indexed canonical spec-chat user-story document at a target-defined path.

Each story has a stable anchored ID and links directly to the canonical spec anchors that define its detailed behavior.

A target defines one stable story-ID prefix and combines it with the origin Linear number and a two-digit per-issue ordinal.

The Linear number records origin only; later issues may evolve the story without changing its ID.

Story IDs are never renamed, renumbered, or reused.

The issue shaper selects the next unused ordinal for that issue by reading the canonical story document; no registry or allocation service exists.

Retirement preserves the anchored story in place with status retired and an optional successor story link.

An active story record contains only stable ID, short title, status, one user-voiced story sentence, exact spec-anchor links, and origin Linear issue.

A retired record may additionally contain one successor story link.

Product-surface sections provide grouping and Git provides history; records carry no acceptance criteria, QA fields, owners, metrics, dates, or implementation detail.

The legacy QA-derived Markdown catalog is candidate source material for one curated migration, never a mechanically converted authority.

Migration groups bug-level and duplicate entries into durable user outcomes.

Migration removes QA evidence, verdicts, acceptance-criterion dumps, proven counts, and generated QA status.

Retained stories receive stable IDs and exact spec links, then undergo direct spec-chat review.

The legacy spec/user-stories.md file is deleted only after the curated spec-chat replacement is approved.

The optional control panel discovers and renders the story document through the same read-only spec path as every other spec.

There is no user-story mirror, publisher, synchronization state, or external-document failure gate.

There is no separate user-story service, database, control-panel backend, mandatory graph, or hand-maintained visualization dataset.

A small journey graph or coverage view may be added only when it materially improves comprehension and derives from the story document itself.

The issue shaper updates the story document during shaping when approved work creates, changes, or retires a user outcome.

The story update and linked behavior specs remain on the same evolving draft PR before final shaping review.

The fresh shaping reviewer verifies story-to-spec alignment and blocks readiness on a material mismatch.

Implementers, code reviewers, QA roles, acceptance helpers, and merge helpers consume the pinned story revision but do not maintain it.

After merge, the control panel reflects the story document from main through normal spec discovery, with no publication job.

Full-pipeline QA consumes a pinned catalog revision and may produce separate coverage, evidence, and verdict artifacts that reference stable story identifiers.

Issue-level QA verifies only active story IDs changed or claimed by the shaped issue.

An explicit full-pipeline QA run verifies every active story at one pinned story and spec revision.

Retired stories are excluded, while missing or unexecutable active stories are reported as not evidenced rather than silently skipped.

No suite tags or secondary test catalog select stories.

QA capture and QA review do not author or silently rewrite product-story truth.

QA and evidence

QA consumes only a clear code-review artifact for the same exact HEAD.

Before capture, QA capture records a small per-criterion evidence plan selecting the minimum honest proof set.

Screenshots are the default proof for stable states.

One screenshot or a concise ordered group may prove a stable state or state sequence when each image has a distinct purpose.

Video is required only when motion, sequence, timing, media playback, or interaction cannot be proved honestly with still images.

Do not capture both screenshots and video unless each format proves a distinct criterion or risk.

Evidence records exact app identity, runtime, HEAD, viewports, fixtures, artifacts, and per-criterion evidence rationale.

A fresh QA reviewer visually inspects the artifacts and grades every criterion pass, fail, or not evidenced.

QA capture produces artifacts and a manifest; a deterministic publication helper updates the evidence site and its index.

Routine publication is incremental; full-archive repair is explicit and never the normal readiness path. Target specs own concrete service-level objectives.

The evidence site is the operator's acceptance-review surface for QA artifacts and the fresh QA reviewer's satisfied or blocking verdict.

Every issue entering Awaiting Accept has a served evidence card, including backend-only work, and readiness is not claimed until the card is readable for the exact issue, PR, HEAD, verdict, story IDs, and criterion coverage.

Every acceptance packet requires a satisfied fresh QA-review verdict before it is offered for operator acceptance.

The operator reviews the served QA artifacts and verdict, then explicitly accepts or rejects; a QA-review verdict never substitutes for operator acceptance.

Evidence publication and QA verdict changes are meaningful gate changes: the orchestrator updates stream status and notifies its parent, then Fable reconciles the whole-operation dashboard.

QA workers never directly mutate the evidence-site index or whole-operation dashboard.

Backend-only work skips browser capture, screenshots, video, and app boot unless the contract itself requires them.

A deterministic helper assembles the clear code-review verdict, exact HEAD, affected story IDs, spec criteria, real unskipped validation, and contract-check outputs into a nonvisual packet.

A fresh QA reviewer checks only nonvisual evidence sufficiency and per-criterion coverage, without repeating code review.

The evidence site serves the backend packet and QA verdict through the same operator acceptance path.

Evidence hosting remains target-repo operations; octo-lite adds no access-control, redaction, retention, or recurring broken-link subsystem.

Publication performs one exact-card and linked-artifact readback before readiness; no periodic evidence sweeper is required.

Known secrets and credentials are never intentionally published.

Messaging

Every Herdr message uses herdr-say, which attempts transport at a verified safe prompt or durably queues without acting on a modal.

herdr-say and herdr-drain never infer composer state, consumption, or delivery from pane transcript text.

A send or Enter transport attempt leaves a message pending; retry reuses the same message ID, retries transport only, and never re-pastes text that already landed.

A successful queued transport persists the pending durable status before removing its inbox retry item; a persistence failure retains the retry item and the queued status.

Visible, injected, pending, or queued TUI text is not acknowledgment.

Commands, operator rulings, ownership transfers, and blocking questions require explicit recipient acknowledgment before dependent action.

Acknowledgment has no arbitrary timeout because dependency, not duration, determines blocking.

Completion requires the named durable artifact or observable outcome, not acknowledgment alone.

An info-kind message stays best-effort transport and never claims acknowledged delivery.

Operator supervision

Fable is the sole writer of the reconciled whole-operation view but does not ingest every raw log.

Upward status leads with product outcome, current gate, blocker, and next operator action; tracker and run identifiers are secondary.

Proactive operator messages are limited to required decisions, blockers, failures, and meaningful gate or outcome changes.

A target operations view becomes primary only when it is accurate and digestible across all workstreams; until then Fable supplies the complete operation update.

Each stream uses one parent-owned brief.md and one child-owned status.md in the operator control directory.

Status compacts upward one parent at a time, and leaf workers do not all message Fable.

One session-durable host timer invokes operator-say sweep throughout the operator lifecycle, including idle periods.

A wake runs deterministic exact checks and may spawn one fresh Sonnet reconciler over the compact normalized snapshot.

A wake with no delta is a cheap no-op and does not remove the timer.

The human may converse directly with an orchestrator, and stream-local rulings apply after durable status capture while cross-stream effects wait for Fable reconciliation.

Operator handoff

Only an explicit operator request starts a Fable-to-Fable handoff.

The outgoing Fable writes one concise immutable Markdown handoff at handoffs/<zero-padded-revision>.md containing objective, stream tree, exact facts, pending decisions, authorities, blockers, control path, timer, and reply routes.

The fresh successor loads canonical instructions, reconciles the handoff against source systems, and declares readiness before cutover.

One operator-owner.toml stores schema version, exact owner session ID, handoff revision, and control-directory identity.

A locked expected-owner and revision check plus atomic TOML rename is the sole authority commit.

The timer and top-level child notifications resolve the owner through operator-say, so the next message observes the same atomic commit.

The old operator becomes read-only at commit and may only forward late direct messages until cleanup.

A dead or ambiguous owner record has no agent-callable recovery path; only an explicit manual operator takeover outside role authority may replace it.

No probe, self-asserted authorization, timeout, or automatic failover transfers authority; a blocked owner record stays blocked until manual takeover.

Workspace and retry

One clean control clone supplies shared Git objects to fresh per-pass worktrees.

Targets may reuse native safe download caches while mutable dependency trees and build outputs remain worktree-local.

Target-owned bootstrap fails fast on wrong repo, base, remote, tools, dependencies, environment, database access, disk health, or validation capability.

Before spawn, check disk health, isolated worktree, conflicting ownership of branch, fixture, port, or deployment resource, and active provider overload; unsafe work waits.

State-mutating QA verifies starting state, avoids fixtures under concurrent mutation, and records its fixture; read-only QA may share stable fixtures.

Completed worker worktrees are removed only after durable handoff, commit, and push state exists, while dirty worktrees stop for inspection.

A completed read-only reconcile worktree is removed only after its result is bound and the receipt is durably persisted, while a failed, diverged, or dirty reconcile worktree is preserved for inspection.

Only failed or interrupted passes add a compact failure record with exact attempt, original error, category, useful work, owner, and next action.

The category is exactly one of input or contract, environment, provider or session, implementation or validation, or external dependency.

Timeout alone is not failure and cannot launch duplicate work.

The owner inspects and preserves useful work before launching a fresh retry instance.

A fleet outage requires the original error, workspace inspection, a fresh exact-model probe, and lane, session, account, provider, or fleet classification.

Artifact formats

octo-lite and participating target repos use spec-chat HTML for canonical human-reviewed domain behavior and ADRs.

Markdown remains canonical for agent-native instructions and lifecycle-bound human context.

TOML is the default for octo-lite-owned machine configuration and current control state.

JSONL is the append-only workflow telemetry format.

JSON or another native format remains only at an external tool, API, schema, or target-consumer boundary.

Screenshots and necessary video remain native media, and generated HTML views remain non-authoritative projections.

Machine state and events carry schema versions, TOML updates atomically, and JSONL appends under a small lock.

Format never determines authority.

Lightweight tracing

Tracing aligns to Langfuse session, trace, and observation concepts at the data-contract level only.

No Langfuse deployment, SDK, collector, exporter, service, or trace database is installed.

Each bounded stream may have one append-only trace.jsonl in its existing control directory.

A session is one bounded orchestrator stream or release train.

A trace is one issue-delivery execution.

An observation is one meaningful workflow boundary such as bootstrap, pass completion, verdict, retry, transition, accepted control message, merge, or deployment fact.

Only deterministic workflow and named gate helpers write traces, and language-model agents never write them directly.

Records contain portable IDs, timing, status, compact role and issue attributes, and artifact references but no raw prompts, responses, logs, transcripts, secrets, evidence bodies, or product data.

A trace inherits its stream control-directory lifecycle and has no separate archive, retention timer, or pruning system.

Runtime append failure writes one trace degraded status warning and continues without retrying or blocking gates.

One path-filtered CI checker validates generated happy, failure and retry, and invalid traces without mutating external systems.

Trace conformance does not replace spec-driven TDD or target-repo tests.

Interfaces and contracts

Role launch

octo-launch is the sole mechanism that bootstraps, verifies, and runs a role pass; a Claude Workflow performs deterministic gating over an already-completed, receipt-bound pass result and never spawns a worker session itself.

The launch helper performs final exact Linear and PR and HEAD readback before dispatch.

It provisions a fresh contained worktree from the declared base and starts the exact role, model, effort, mode, and skill set.

It captures the exact provider session ID and requires a compact bootstrap acknowledgment before mutation.

Workspace containment is checked exactly at provisioning and child bootstrap, not before every command.

The receipt binds spawn, parent and reply route, role, runtime, repo, instructions, issue, spec, PR, topology, starting HEAD, and required prior gates.

The initial launch context and receipt state only execution location, operator loopback reachability, and required review delivery path.

Unless launch context explicitly confirms operator loopback access, every role treats loopback as unreachable and never presents a localhost URL as usable operator access.

Durable access behavior belongs in octo-lite specs and roles, durable target-specific conventions belong in target AGENTS.md, current access facts belong in the receipt or stream status, and temporary routes belong only in stream status.

Role, declared purpose, and launcher-resolved capability set are one validated combination checked before any worktree creation or provider bootstrap, and an invalid combination fails closed leaving no worktree, receipt, provider session, or partial launch state.

A shaping-review purpose is admitted only for role shaping-reviewer, or role orchestrator whose launcher-resolved capability set includes shaping; every other role, and an orchestrator without resolved shaping capability, is rejected.

A delivery purpose is admitted only for role implementer, code-reviewer, qa-capture, or qa-reviewer; every other role is rejected.

Every executable launcher entrypoint, whether a standalone bootstrap from an already-persisted receipt, a standalone acknowledgment verification, or a bootstrap composed into a full launch or reconcile pass, revalidates the receipt against its manifest shape and its launch revision at the shared boundary immediately after receipt readback and before the provider runner is invoked, and again before marking any receipt ready or bootstrap-verified, so a manifest-shape violation, a pre-fix or altered invalid combination, or a launch-revision mismatch is rejected with no provider process started and no receipt marked ready or bootstrap-verified.

Manifest-aware revalidation admits an octo-lite-pass receipt only when it satisfies the shaping-review or delivery role-purpose-capability matrix, an octo-lite-reconcile receipt only for role reconciler carrying no pass purpose, and a generic persistent launch receipt only for role meta-operator or orchestrator carrying no pass purpose, each preserving its existing ready semantics; a pass purpose injected onto a non-pass receipt, a role substituted for the receipt's manifest shape, or an unknown manifest type is rejected.

A canonical launcher-produced receipt binds a [shaping] table with repo, PR number, and exact starting HEAD from the freshly verified PR only for role orchestrator whose launcher-resolved capability set includes shaping and purpose shaping-review; every other role, an orchestrator without resolved shaping capability, or any other purpose carries no shaping authority.

An OpenAI review pass needing live GitHub or Linear reads bootstraps read-only, then resumes with explicit workspace-write sandbox and network access, and launch rejects any read-only review-pass worktree HEAD or status change found after bootstrap or after the resumed pass.

Every OpenAI resume call selects its sandbox through -c sandbox_mode=... config, never the top-level -s flag, because the installed CLI's resume subcommand rejects -s while its top-level exec bootstrap still accepts it.

Launch admits a shaping-review purpose only for Linear state Ideas, Todo, Shaped, or In Progress, and admits a delivery purpose only for Linear state Shaped, Todo, or In Progress; every other state, including Awaiting Accept and terminal states, is rejected for both purposes.

Admitting In Progress for both purposes lets shaping re-review run after material reconciliation and lets implementation, code review, re-review, and QA review workers launch while the stream orchestrator holds In Progress, without weakening the exact shaping-verdict checks a delivery purpose still requires.

Edge cases

A modal or question dialog causes herdr-say to queue rather than inject or submit.

A changed implementation HEAD that changes behavior or contract invalidates review and QA; packet-only or evidence-wording changes do not.

A test command that silently skips required database coverage is not green validation.

A generated packet that is not readable on the served evidence site is not ready for acceptance.

A PR-owning GitHub identity does not block review because the canonical gate is the exact-HEAD verdict comment, never a fabricated formal review state.

A malformed trace blocks the relevant CI contract check but does not retroactively invalidate authoritative gate artifacts.

A missing or ambiguous owner record blocks every operator-owned mutation until manual operator takeover outside role authority; no role or command may self-authorize recovery.

Constraints

Prompts, roles, skills, and receipts remain extremely concise and use progressive disclosure.

Skills use no em dash or en dash characters.

Generic octo-lite mechanics are not copied into target repos.

Turbo-specific paths, credentials, fixtures, states, and dashboard mechanics do not live in generic octo-lite prompts or skills.

The optional hosted control panel is read-only and never becomes a second control plane.

Source systems and control artifacts remain authoritative if every generated view is unavailable.

Non-goals

Building an octo-lite service, scheduler, event bus, tracker mirror, orchestration database, or watcher fleet is out of scope.

Building a policy compiler, role registry service, giant spawn manifest, or inheritance engine is out of scope.

Running Langfuse or OpenTelemetry infrastructure is out of scope.

Building a broad Cartesian workflow conformance matrix is out of scope.

Making transcript replay, memory, pane pixels, or queued text a correctness dependency is out of scope.

Building an agent-callable dead-owner recovery command, liveness-probe authority, or cryptographic operator-authentication service is out of scope.
