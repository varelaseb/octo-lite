octo-lite delivery lifecycle

Specifications / operating model / delivery lifecycle

Canonical shaping, TDD, delivery, Linear, release, story, QA, and acceptance behavior.

Shaping readiness gate

Shaping contract

Shaping begins from the complete current Linear issue for Linear-tracked work.

The issue shaper grills one unresolved question at a time with a recommended answer and records each resolved behavior immediately.

Shaping is not complete without an aligned Linear issue, canonical spec, evolving draft PR, and final fresh shaping-review verdict.

The shaping reviewer reads canonical specs and ADRs, relevant conversation logs through a pinned cutoff, Linear, repo instructions, the same draft-PR readiness HEAD, and the TDD contract.

Linear Shaped is the canonical readiness state and is set only after a clear final shaping verdict.

Prior explicit operator intent authorizes Shaped, recorded as a durable operator-intent decision, and the clear final shaping-review verdict on the draft PR is the separate readiness proof; the transition is act-then-notify with a prepared rollback.

The operator notification presents human-sensical context only: a plain-language what-changed-since-your-last-look summary plus the rendered pages, and never presents or requests an exact head or hash.

Tooling stamps the exact head automatically at action time and records that binding on the PR and status surfaces, never inside the operator notification.

The verified issue orchestrator sets Shaped on a live clear-verdict readback under prior explicit operator intent, then notifies with a prepared rollback; shaped-scope authorization never substitutes for the distinct acceptance step that precedes merge.

A verified issue orchestrator in shaping mode may move Ideas or Todo to Shaped only after live readback of one canonical clear shaping verdict at the exact PR HEAD and a linear:<issue>:<fingerprint> binding for the exact current Linear issue.

Existing delivery and operator boundaries remain unchanged.

The final shaping verdict is a durable artifact on the evolving draft PR, not a separate readiness file.

The verdict binds the relevant Linear content fingerprint, PR and shaping HEAD, spec and ADR blobs, conversation-log references and cutoff, clear or blocking result, and reviewer receipt reference.

Linear links the PR verdict without duplicating its exact-revision proof.

Implementation launch refetches Linear, the PR, HEAD, specs, logs, and verdict; any mismatch returns the issue to shaping.

Shaping and code review each publish one concise machine-readable PR verdict comment through a deterministic helper.

The comment records review type, clear or blocking verdict, exact HEAD, bound inputs, findings, and reviewer receipt reference.

No formal GitHub Review API operation, reviewer-identity preflight, check run, or fallback tree is required for workflow correctness.

Documentary gaps return to the shaper, missing judgments return to the operator grill, and material reconciliation requires another fresh shaping review.

The implementer treats the spec as detailed behavioral law and Linear as high-level decision, context, identity, non-goal, and status law.

Conversation logs are shaping-review evidence, not implementation instructions.

If Linear high-level law conflicts with canonical spec or ADR law, neither silently wins: stop implementation and return to shaping until both and the evolving PR agree.

Prompt TDD sequence

Prompt TDD

Every prompt, role, or skill instruction change starts with a behavior contract naming exact inputs, the required observable decisions, actions, or fields, forbidden behavior, and acceptable proof.

A fresh model session runs the smallest scenario against the actual unchanged instructions first; only a genuinely wrong or ambiguous decision is red.

A missing file, module, export, or script is not a valid prompt red.

A different fresh model session runs the same scenario against the changed instructions and judges the observable decision against the contract for green, then narrow regression scenarios run for adjacent behavior the change could shift.

Deterministic tests cover loading, wiring, schema, invariants, and forbidden characters; they judge structure, never prompt judgment quality.

An exact prose snapshot or keyword presence is never behavioral proof.

Red chronology that cannot be honestly reconstructed must be stated as such, never fabricated.

Eval evidence stays a lean one-off scenario record; no eval service or general harness.

Delivery TDD

Code delivery TDD is distinct from prompt TDD, which stays prose only, because a code delivery pass produces durable commits an independent observer re-runs.

The implementer commits a real failing red, a new or changed test plus unchanged production, that genuinely fails the named test, as a durable commit on an isolated delivery branch.

A missing file, module, export, or script is not a valid red.

The implementer then commits the production change that makes the same named test pass, on the same isolated branch.

The failing test is bound by path and content digest, not by name alone, so the same test file with the same content digest present in the red commit is present unchanged in the green commit; the green commit changes production only and never the bound test, and a green that removes, weakens, or edits the red's failing test is rejected.

A separate Read-restricted observer, the dedicated tdd-observer role and never the mutating worker, checks out and runs each committed state and confirms the red commit fails and the green commit passes the same scenario, and worker-authored red or green strings are never proof.

The observer checkout and run happen in an isolated worktree that never disturbs the worker branch, main, or the live repository working directory.

The observer derives its test invocation only from a trusted source, the committed test files themselves and the canonical validation command declared in the target AGENTS.md, and never from any worker-supplied string; worker-controlled command or scenario strings never enter the observer's prompt or execution inputs, only its own trusted-source-derived invocation runs.

The observer's execution inputs, both the red, green, and final-HEAD commit identifiers it checks out AND the test command, are supplied by the host from the journal binding of the worker's committed delivery branch, and never from any worker-authored claim.

The host records the committed branch and its red, green, and final commit ids in the journal, and the observer checks out exactly those host-journalled commits, so a worker-authored commit identifier is never a source of what the observer executes.

Refactor commits after green are allowed, and the exact final pushed HEAD, after any refactor, is independently checked out and executed by the observer and confirmed green before the host push is accepted, so nothing is pushed whose final HEAD is not independently verified.

The final-HEAD observation also confirms the bound test is present unchanged at the final pushed HEAD by the same path and content digest as the red and green bound test, not merely that tests pass, so a green execution alone never accepts a final HEAD whose bound test a refactor weakened, edited, or removed; a final HEAD whose bound test path is absent or whose content digest differs from the bound identity is rejected.

The mutating worker never pushes, and the host pushes the verified branch only after the independent observer confirmation of the red, the green, and the final HEAD, the acknowledgment echo, and a fresh pre-push readback, then confirms the pushed HEAD by a live remote read, gh api or git ls-remote, not a local tracking ref.

A rejection abandons the unpushed branch.

Only the observer re-run of the committed states is proof, and worker-controlled command or scenario strings are never interpolated into the observer's prompt, execution inputs, or verdict so independence cannot be forged.

The smallest failing delivery TDD, the exact tests delivery implements under this model, are named and spec-derived.

  • test_observer_replays_committed_red_then_green re-runs the committed red and green in an isolated worktree and reports red fail then green pass for the same scenario.
  • test_observer_rejects_worker_supplied_observation rejects a worker-authored red or green verdict string as proof.
  • test_observer_inputs_exclude_worker_strings proves no worker-supplied command or scenario string enters the observer prompt or execution inputs.
  • test_observer_commit_inputs_are_host_journalled proves a worker-claimed commit id that differs from the host-journalled committed branch commits is rejected, and the observer checks out only host-journalled commits.
  • test_observer_command_source_is_host_trusted proves the observer's test command is the canonical validation command from the target AGENTS.md supplied by the host, and a worker-authored command string is rejected rather than executed.
  • test_invalid_red_missing_file_rejected rejects a red that is only a missing file, module, export, or script.
  • test_invalid_red_that_does_not_fail_rejected rejects a red commit whose named test passes.
  • test_test_identity_binding_green_may_not_weaken_red rejects a green that removes, weakens, or edits the red's failing test by path and content digest.
  • test_push_ordering_host_push_only_after_observation_echo_readback proves the host push runs only after observation, ack echo, and the fresh pre-push readback, and the worker never pushes.
  • test_rejection_abandons_unpushed_branch proves a rejected pass leaves its isolated branch unpushed with no destructive reset.
  • test_final_head_independently_verified_green proves the exact final pushed HEAD, after any refactor, is independently executed green by the observer before the push is accepted.
  • test_final_head_binds_test_identity proves the final-HEAD observation confirms the bound test is present unchanged at the final pushed HEAD by the same path and content digest as the red and green bound test, and rejects a final HEAD whose bound test a refactor weakened, edited, or removed.
  • test_push_readback_uses_live_remote proves the pushed-HEAD confirmation reads the live remote through gh api or git ls-remote, never a local tracking ref.

Exact-head delivery loop

Delivery loop

This is the single canonical delivery entry gate: delivery enters only from the Shaped readiness gate, loop fire performs the one mechanical Shaped -> Todo transition making Todo the first delivery step, and delivery then proceeds through Todo, In Progress, and Awaiting Accept.

The octo-lite-loop skill, the launcher launch-linear-state-gate, and profile/AGENTS.md all defer to this definition and never restate a divergent entry rule.

Every fix changes HEAD and requires a fresh code review before QA.

The exact final pushed HEAD of a delivery pass, after any refactor commit, is the HEAD the independent observer executes and confirms green under delivery-tdd-final-head-verification, and the observer also confirms the bound test is present unchanged at that final HEAD by the same path and content digest as the red and green bound test under delivery-tdd-final-head-test-identity, so no delivery pass pushes a HEAD the observer did not independently verify or whose bound test a refactor weakened, edited, or removed.

At most three fresh review and fix cycles may run before the issue returns to shaping.

Behavior or contract changes invalidate review and QA while packet-only wording changes do not.

Actual acceptance of delivered work stays a human-gated operator carve-out gated on the human instruction: no agent infers human acceptance, decides acceptance as the human, or merges unaccepted work, and no worker delivery role merges at all, while on an explicit human acceptance instruction the operator agent executes the merge against the operator-owned acceptance artifact for the same verdict and HEAD, and every earlier gate on the way to Awaiting Accept runs act-then-notify with prepared rollback under prior explicit operator intent.

Topology selection and integration

Branch and release topology

The same draft PR continues from shaping through implementation.

The shaping readiness HEAD remains immutable evidence while implementation advances the PR HEAD.

Sequential, stacked, or parallel issue delivery is chosen by the accountable epic orchestrator from actual dependencies, collision risk, review cost, and throughput, never by a fixed programmatic default.

The train manifest records the current chosen topology and revision; it does not select or enforce topology.

Member PRs integrate into the declared carrier or predecessor, never directly into main; the carrier is the train's sole final-main PR.

After member integration, the orchestrator proves reviewed-HEAD ancestry and effective diff before preserving downstream gates.

The stream orchestrator has standing authority to select and revise main, stacked-branch, or integration-train topology without per-change approval.

Topology changes escalate only when ship grouping, product scope, or material risk changes.

Authorized PR retargeting preserves stacking and release trains but must record topology revision, merge base, ancestry, effective diff, and invalidated downstream gates.

A multi-PR train has one carrier PR and one machine-readable manifest listing member Linear issues, PRs, pinned HEADs, order, topology revision, integration branch, target, owner, and gate state.

The train manifest owns topology only and never duplicates product scope or lifecycle truth.

Linear lifecycle authority

Linear lifecycle

Gate-critical reads use one exact Linear helper returning the complete issue, stable identity, revision, and current state.

A mechanical transition compares expected state, mutates once, reads back, updates stream status, and notifies the parent before reporting success.

A partial transition failure retries only unfinished idempotent steps and never blindly repeats a completed state mutation.

Stream orchestrators may perform only Todo -> In Progress -> Awaiting Accept through that helper.

Loop fire is the one mechanical Shaped -> Todo transition through that helper, moving shaped-not-started work into Todo as the first delivery step, and it never moves Shaped directly to In Progress.

Acceptance of delivered work, preproduction promotion, and live traffic shifts are the three human-gated operator carve-outs gated on the human instruction, so on an explicit human instruction the operator agent executes the resulting acceptance record, the bound merge of the accepted work, the preproduction promotion, or the traffic shift and never hands the mechanical action back to the human, while no agent self-authorizes or infers acceptance, a preproduction promotion, or a traffic shift, and no agent merges unaccepted work; the staging transition runs act-then-notify with prepared rollback under prior explicit operator intent as an automatic post-merge step, and the other lawful stops are operator-held access and legally binding irreversible actions without rollback.

Acceptance inside a stack or release train remains an acceptance-ledger fact while Linear stays Awaiting Accept until the commit reaches main.

Generic octo-lite defines only that post-acceptance deployment and promotion states are durable, operator-authorized, and evidence-backed; the exact state names and the concrete facts that trigger each one are target-owned Linear state mapping, defined in the target repo, never in generic octo-lite.

Sweep operator-gate surfacing

Each operator sweep run reads the current Linear state of every issue a stream carries, not one bound issue, so no carried issue state is ever unread.

A stream's carried issues are derived from its structured declarations, because real production streams have no sources.toml, so a single explicit binding source would leave carried issues unread.

The carried set is the deduplicated union, in priority order, of the sources.toml [linear] issue or issues, every review or delivery *envelope*.toml top-level issue, and the status.md anchored Issues: line, with issue tokens matched case-insensitively and normalized to the upper canonical form.

Only the single anchored Issues: line of status.md is parsed for issue tokens, never the free prose of the document, so unrelated roadmap issue mentions are never surfaced as carried operator-gated work.

A Linear state is operator-gated when it awaits an operator act: the acceptance state Awaiting Accept plus every operator-owned promotion state in the target Linear mapping, and it excludes the terminal live state, the accepted-and-merged transient, and every agent-owned or mechanical state.

Any carried issue in an operator-gated state emits, at the very top of the sweep delta before all other content, one loud line naming the issue id, the state, and a one-line operator action.

The surfaced operator-gate output, meaning the loud lines on sweep stdout and in the delta result, is exempt from every suppression, vestige, consumed-ask, and unchanged-fingerprint class, so an unchanged sweep still surfaces every current gate and no heuristic ever silences that surface.

The loud operator pane ping is edge-triggered and distinct from that always-on surface: it fires once when the operator-gate set becomes non-empty or changes, re-arms when the set clears, and never re-pings the operator for an unchanged set, so a gate parked for hours does not re-flood the operator every cycle.

The edge marker fingerprinting the sorted gate set persists only when the pane ping is delivered or queued at a modal, so a hard send failure leaves it unwritten and the next sweep re-pings rather than silently dropping a new gate.

A stream carrying an operator-gated issue classifies waiting-on-operator regardless of transcript activity, never active or suspected-stuck.

A stream with an open unanswered operator-facing question surfaces the same loud top line naming the stream and the verbatim ask, the lane operator-ask path stamps that wait mechanically, the operator answer relay clears it, and detection is modal-safe and never scrapes pane transcript text.

Each sweep probes every non-closed stream's herdr agent, targeted by the stream name the spawn path assigned as the agent name, through the grammar-stable non-blocking agent get probe, and a stream whose agent is gone with no live transcript activity classifies dead, a distinct class ranked between closed and waiting that stops a dead lane from perpetually re-flagging suspected-stuck.

A stream whose status.md anchored Outcome: line records superseded-closed, matched case-insensitively on that anchored line only and never in free prose, classifies closed and is excluded from suspected-stuck and dead flagging, while any operator gate a closed or dead stream carries still surfaces loudly and still classifies waiting-on-operator.

A missing herdr binary, an unreachable herdr socket, an unrecognized probe payload, or a probe timeout degrades the watchdog gracefully to the time-based classification with a noted error and a bounded per-stream probe latency, and watchdog failure never crashes the sweep.

Story, specification, and QA contract

User-story surface

The user-story catalog is a product-surface artifact and a QA verification contract for full-pipeline checks.

The story record is not QA evidence and does not embed screenshots, videos, verdicts, proof links, or acceptance state.

A participating target owns one indexed canonical spec-chat user-story document at a target-defined path.

Each story has a stable anchored ID and links directly to the canonical spec anchors that define its detailed behavior.

A target defines one stable story-ID prefix and combines it with the origin Linear number and a two-digit per-issue ordinal.

The Linear number records origin only; later issues may evolve the story without changing its ID.

Story IDs are never renamed, renumbered, or reused.

The issue shaper selects the next unused ordinal for that issue by reading the canonical story document; no registry or allocation service exists.

Retirement preserves the anchored story in place with status retired and an optional successor story link.

An active story record contains only stable ID, short title, status, one user-voiced story sentence, exact spec-anchor links, and origin Linear issue.

A retired record may additionally contain one successor story link.

Product-surface sections provide grouping and Git provides history; records carry no acceptance criteria, QA fields, owners, metrics, dates, or implementation detail.

The legacy QA-derived Markdown catalog is candidate source material for one curated migration, never a mechanically converted authority.

Migration groups bug-level and duplicate entries into durable user outcomes.

Migration removes QA evidence, verdicts, acceptance-criterion dumps, proven counts, and generated QA status.

Retained stories receive stable IDs and exact spec links, then undergo direct spec-chat review.

The legacy spec/user-stories.md file is deleted only after the curated spec-chat replacement is approved.

The optional dashboard discovers and renders the story document through the same read-only spec path as every other spec.

There is no user-story mirror, publisher, synchronization state, or external-document failure gate.

There is no separate user-story service, database, dashboard backend, mandatory graph, or hand-maintained visualization dataset.

A small journey graph or coverage view may be added only when it materially improves comprehension and derives from the story document itself.

The issue shaper updates the story document during shaping when approved work creates, changes, or retires a user outcome.

The story update and linked behavior specs remain on the same evolving draft PR before final shaping review.

The fresh shaping reviewer verifies story-to-spec alignment and blocks readiness on a material mismatch.

Implementers, code reviewers, QA roles, acceptance helpers, and merge helpers consume the pinned story revision but do not maintain it.

After merge, the dashboard reflects the story document from main through normal spec discovery, with no publication job.

Full-pipeline QA consumes a pinned catalog revision and may produce separate coverage, evidence, and verdict artifacts that reference stable story identifiers.

Issue-level QA verifies only active story IDs changed or claimed by the shaped issue.

An explicit full-pipeline QA run verifies every active story at one pinned story and spec revision.

Retired stories are excluded, while missing or unexecutable active stories are reported as not evidenced rather than silently skipped.

No suite tags or secondary test catalog select stories.

QA capture and QA review do not author or silently rewrite product-story truth.

QA proof selection and acceptance

QA and evidence

QA consumes only a clear code-review artifact for the same exact HEAD.

Before capture, QA capture records a small per-criterion evidence plan selecting the minimum honest proof set.

Screenshots are the default proof for stable states.

One screenshot or a concise ordered group may prove a stable state or state sequence when each image has a distinct purpose.

Video is required only when motion, sequence, timing, media playback, or interaction cannot be proved honestly with still images.

Do not capture both screenshots and video unless each format proves a distinct criterion or risk.

Evidence records exact app identity, runtime, HEAD, viewports, fixtures, artifacts, and per-criterion evidence rationale.

A fresh QA reviewer visually inspects the artifacts and grades every criterion pass, fail, or not evidenced.

QA capture produces artifacts and a manifest; a deterministic publication helper updates the evidence site and its index.

Routine publication is incremental.

Full-archive repair is explicit and never the normal readiness path.

Target specs own concrete service-level objectives.

The evidence site is the operator's acceptance-review surface for QA artifacts and the fresh QA reviewer's satisfied or blocking verdict.

Every issue entering Awaiting Accept has a served evidence card, including backend-only work, and readiness is not claimed until the card is readable for the exact issue, PR, HEAD, verdict, story IDs, and criterion coverage.

Every acceptance packet requires a satisfied fresh QA-review verdict before it is offered for operator acceptance.

The operator reviews the served QA artifacts and verdict, then explicitly accepts or rejects; a QA-review verdict never substitutes for operator acceptance.

Evidence publication and QA verdict changes are meaningful gate changes: the orchestrator updates stream status and notifies its parent, then Fable reconciles the whole-operation dashboard.

QA workers never directly mutate the evidence-site index or whole-operation dashboard.

Backend-only work skips browser capture, screenshots, video, and app boot unless the contract itself requires them.

A deterministic helper assembles the clear code-review verdict, exact HEAD, affected story IDs, spec criteria, real unskipped validation, and contract-check outputs into a nonvisual packet.

A fresh QA reviewer checks only nonvisual evidence sufficiency and per-criterion coverage, without repeating code review.

The evidence site serves the backend packet and QA verdict through the same operator acceptance path.

Evidence hosting remains target-repo operations; octo-lite adds no access-control, redaction, retention, or recurring broken-link subsystem.

Publication performs one exact-card and linked-artifact readback before readiness; no periodic evidence sweeper is required.

Known secrets and credentials are never intentionally published.

Edge cases

A changed implementation HEAD that changes behavior or contract invalidates review and QA; packet-only or evidence-wording changes do not.

A test command that silently skips required database coverage is not green validation.

A generated packet that is not readable on the served evidence site is not ready for acceptance.

A PR-owning GitHub identity does not block review because the canonical gate is the exact-HEAD verdict comment, never a fabricated formal review state.
