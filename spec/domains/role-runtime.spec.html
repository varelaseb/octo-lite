Role and model topology

The complete LLM role roster is meta-operator, orchestrator, shaping-reviewer, implementer, code-reviewer, qa-capture, qa-reviewer, reconciler, and the delivery tdd-observer.

The issue orchestrator performs shaping mode by resolving the issue-shaper skill in its own persistent session, never a second persistent role or tab.

Shaping produces and commits the durable contract, the high-level issue, canonical specs, ADRs, and user stories, and publishes it to the evidence or spec page before any implementer pass, and a fresh implementer works only off that committed spec while the reviewer checks code against it, so an implementer never authors the spec.

Every fix uses a fresh implementer instance and every re-review uses a fresh code-reviewer instance rather than separate role contracts.

Every nested stream owner uses the same orchestrator contract.

Every issue has one and only one dedicated Opus issue orchestrator.

A multi-issue epic adds a separate Opus epic orchestrator above its issue orchestrators, producing two Opus layers below Fable.

Fresh workers report to their issue orchestrator; issue orchestrators report to the epic orchestrator; the epic orchestrator reports upward to Fable or its declared parent.

A parent flattens relevant effective rules into the child brief; the child acknowledges them only at spawn or after a material brief revision.

Acceptance recording, merge, evidence publication, Linear transition, preproduction promotion, and live traffic shift are deterministic operator-authorized helpers rather than LLM roles.

The dashboard displays deterministic gates and helpers separately from LLM roles.

The exact model, effort, and service tier for every role, including the delivery tdd-observer, are normative in the role and model map diagram above; the constraints below never restate that mapping.

The meta-operator runs in auto permission mode.

Fable 5 uses its default 1M context without a redundant [1m] suffix.

An unavailable Fable 5 fails visibly and is never silently replaced by Opus or another model.

Stream and subordinate orchestrators run in auto permission mode.

An unavailable orchestrator model fails visibly and is never silently replaced by another version or family.

Implementation and every fix pass run as a fresh exact instance of the mapped implementer role, never reused across passes.

Code review and every re-review run as a fresh exact instance of the mapped code-reviewer role.

The final post-grill shaping review runs as a fresh exact instance of the mapped shaping-reviewer role.

QA capture runs as a fresh exact instance of the mapped qa-capture role.

QA review runs as a fresh exact instance of the mapped qa-reviewer role.

Each reconciliation sweep runs as a fresh exact instance of the mapped reconciler role that never falls back.

Deterministic helpers perform exact checks and provide one compact normalized read-only snapshot; Sonnet classifies only changed, missing, stale, contradictory, or needs_fable.

The reconcile gateway reads the exact snapshot file bytes from a regular contained path and hashes them, rejecting a missing, substituted, corrupt, or digest-mismatched snapshot before any worktree or provider launch.

Reconcile PR facts bind normalized PR state, review decision, and status check rollup outcomes alongside head, base, and branch, so a change in any of them changes the snapshot fingerprint.

Normalized status check facts sort on name, current status, and outcome so duplicate-name checks in any API order, or with differing outcomes or in-progress states, never collapse or reorder the same rollup's fingerprint.

The reconciler's canonical spec and ADR sweep requires every declared canonical path to exist as a readable regular file at the exact control HEAD, and a missing, renamed, unreadable, non-file, or empty resulting source set fails the sweep closed before any snapshot or worktree is produced.

Each target repo root AGENTS.md declares its own canonical spec and ADR paths as exact comma-separated Canonical spec paths and Canonical ADR paths lines, and octo-lite's own AGENTS.md declares its own paths the same way rather than a hardcoded constant.

A target initialization template declares exactly one Canonical spec paths placeholder and exactly one Canonical ADR paths signal in root AGENTS.md alongside the spec-format signal.

Shaping replaces the spec placeholder with exact target-owned canonical paths before readiness, and no role, skill, workflow, or octo-lite dependency is copied into the target.

The canonical sweep fails closed on an absent declaration, a missing, non-blob, empty, duplicate, malformed, escaped, or symlinked declared path, or a path declared under both the spec and ADR label.

Canonical ADR paths: none is the sole explicit no-ADR representation and binds an empty ADR blob list rather than a path.

Any other empty value, sentinel, a mix of none and a path, or a repeated Canonical ADR paths declaration fails the sweep closed, and at least one canonical spec path remains mandatory.

Zero or more than one signal line for either label fails the sweep closed, a prefix lookalike line never counts toward that total, and duplicate paths within one otherwise valid line still fail closed.

A sweep refetches and validates current Linear, GitHub PR, target HEAD, and canonical spec and ADR facts before any unchanged decision or snapshot persistence, and the compared fingerprint binds all of them rather than only declared stream facts.

The sweep captures one target HEAD and passes it as the gateway's expected HEAD, and the gateway compares its own freshly read current repo HEAD against that expected HEAD before validating any source or creating a worktree.

Any HEAD change between capture and gateway comparison, including an unrelated commit that leaves canonical blobs unchanged, fails before any provider launch and persists no snapshot, receipt, result, worktree, or sweep state.

The journalled reconcile binding and the child worktree bind the same expected HEAD.

A snapshot, sweep state, and worktree are persisted only after every bound fact validates; a failed target HEAD read or canonical sweep leaves none of them behind.

The sweep never persists snapshot.md under the final sweep directory before Linear, PR, HEAD, spec, and ADR validation all succeed, and a failing gateway leaves no sweep artifact directory or state behind.

A temporary input required only for digest verification lives outside the final sweep directory and is removed on every failure, while a successful sweep still persists its atomic snapshot, journal entry, result, and no-op state.

The reconcile journal entry binds the snapshot path to the final persisted snapshot.md, never the temporary digest-verification input, and that exact file exists with matching bytes and digest before the reconciler subagent spawns.

Every caught pre-spawn failure, including a journal persistence failure that happens after snapshot persistence, leaves no final sweep snapshot, journal entry, or sweep directory and no pending digest-verification input behind.

The reconciler subagent acknowledgment opens the exact file at the journal-bound snapshot path, hashes its bytes, and refuses the echo when that file is missing or its digest does not match the journal-bound snapshot digest.

Sonnet cannot mutate sources, override deterministic mismatches, investigate open-endedly, or silently resolve ambiguity.

Missing or unparseable input and semantic ambiguity return needs_fable; Fable judges the case.

Orchestrated Claude Workflows are the primary loop runtime rather than an adapter target, and every worker role runs as a subagent of the owning session's workflow.

Every worker role migrates to a Claude Workflow subagent, including implementer, qa-capture, reconciler, and the delivery tdd-observer; reconciler, the tdd-observer, and the review roles run under a Read-restricted subagent type, OpenAI review roles run through the codex relay subagent, and only the meta-operator and the epic and issue orchestrators remain persistent host sessions.

The implementer worker commits the red and green on an isolated branch and never pushes, and the host gates the push after independent observer verification, the acknowledgment echo, and the fresh pre-push readback.

The tdd-observer is a dedicated Read-restricted delivery role, never the mutating worker, that checks out each committed state and the final HEAD in an isolated worktree and runs the trusted-source-derived invocation to confirm the red fails, the green passes, and the final pushed HEAD is green; it never mutates the worker branch, main, or the live repository working directory, and it never pushes.

The tdd-observer derives its invocation only from the committed test files and the canonical validation command in the target AGENTS.md, and no worker-supplied string enters its prompt or execution inputs, so its verdict cannot be forged.

The tdd-observer receives its commit identifiers and command from the host journal binding of the worker's committed delivery branch, never worker-authored, and it checks out exactly the red, green, and final commit ids the host journalled.

The reconciler has two execution modes selected by the driving context, not one migrated path that replaces the other, and ruling-48 fixes this bifurcation as canonical.

The headless subprocess reconcile runs when scripts/operator-sweep fires under a systemd --user timer with no Claude or Workflow runtime and no owning session, so it drives the reconciler through the retained subprocess provider-argv relay (octo_lite.launch.prepare_reconcile_launch then octo_lite.launch.run_reconcile_launch).

The session Workflow reconcile runs when an owning Claude session drives the loop, so the reconciler spawns as a Read-restricted Workflow subagent that the session binds and verifies through octo_lite.launch.bind_reconcile_workflow_journal, octo_lite.launch.verify_reconcile_workflow_ack, and octo_lite.launch.bind_reconcile_workflow_result.

The headless-timer constraint is that a systemd --user timer cannot spawn a Workflow subagent because it has no Workflow runtime and no session, so the Workflow-native reconciler is for session contexts only and can never serve the headless timer.

Because the timer has no other path, the subprocess provider-argv relay is retained and is not a bypass, and it must not be removed; ruling-47 and ruling-48 voided the reconcile-relay removal for exactly this reason, and the two modes coexist rather than the Workflow mode superseding the subprocess mode.

Each roles/<role>.md is the sole canonical prose contract for that role.

roles.toml exclusively owns provider, exact model, effort, permission mode, tools, required and conditional skills, and contract path.

Role prose owns responsibilities, authority, stop conditions, inputs, and outputs without repeating machine configuration.

One small resolver combines the canonical role contract and machine mapping before every Workflow agent invocation or Sol relay.

Claude Markdown adapters are generated launcher inputs and diagnostic surfaces, never alternate role-bearing spawn paths.

Codex has no generated custom-agent adapter; Codex custom agents are not part of correctness or fallback for any role.

An OpenAI-backed role pass runs inside a Claude relay subagent that executes one explicit codex exec relay, carrying exact model, effort, service tier, worktree, and the canonical contract as the exec prompt, and returns the codex final message verbatim with a hash the workflow binds; it never summarizes the relay output and never discovers or depends on a Codex custom agent.

The relay fails closed unless the exact effective provider, model, and effort are independently read back from the Codex session's own rollout record before any mutation.

Claude Workflows load canonical role prose verbatim and Sol relays receive the same prose without copied workflow literals.

The launcher loads every required skill plus only matched conditional skills and records the actual resolved set.

The launch receipt records role path and git blob, roles.toml revision, exact runtime values, and resolved skills.

Adapter generation fails if a target platform cannot express a required tool, mode, or contract field.

roles.toml contains no target rules, issue scope, live state, secrets, or operator decisions.

One path-filtered CI dry run proves role blob, map revision, model, mode, skills, target instruction blob, repo, worktree, and starting HEAD reached the spawned session.

If the Workflow runtime cannot read canonical files directly, a deterministic prebuilt Workflow injects the exact role text and source blob without handwritten prompt literals.

A worker instance is never resumed for another pass and --last is prohibited.

A completed orchestrator stream archives its control record and terminates rather than accepting later work.

Fail-closed launch pipeline

Role launch

A Claude Workflow inside the owning orchestrator session spawns, gates, and runs every worker role pass as a subagent; the deterministic admission and revalidation gates run in the workflow-layer helpers, and OpenAI review roles run through a Claude relay subagent that executes the codex exec relay and returns findings verbatim. Persistent sessions are hosts that run these workflows, never subagents spawned by them.

The launch helper performs final exact Linear and PR and HEAD readback before dispatch.

It provisions a fresh contained worktree from the declared base and starts the exact role, model, effort, mode, and skill set.

A worker subagent requires a schema-forced acknowledgment echo of its exact bound inputs before mutation; the host session owns the subagent, so no separate provider session ID handshake is captured for a Claude worker, and only the codex relay subagent returns its proven codex session identity.

Workspace containment is checked exactly at provisioning and child bootstrap, not before every command.

The host provisions the per-lane isolated worktree and runs the loop inside it, so worker workspace identity derives from provisioning by the host and never from a child-supplied or subagent-read claim, the same host-owns-the-trust-root rule as the persistent control root applied at the worktree level.

Each lane provisions its own exclusive worktree pinned to its branch and starting commit, and provisioning refuses the control repository path itself and any path already bound to a different lane or branch, so no worker ever operates the shared control directory and no worktree is ever checked out to a foreign lane branch.

Provisioning emits a host-authored provisioning record stamped with source host-provisioned-worktree carrying schema version, lane, control repo, worktree, worktree root, repo slug, branch, resolved starting head, resolver root, install-check state, and provisioned-at, and a record from any non-host source or missing a field fails closed.

The provisioning record is written to a host control location outside the worktree tree, never inside it, so the verified-clean tree stays clean and idempotent reuse and cleanup are never broken by a tracked or untracked record file, while OCTO_PROVISION_RECORD names that out-of-tree path.

The record is a JSON object with exactly these keys and constraints: schema_version integer 1; source the string host-provisioned-worktree; lane a nonempty string; control_repo the octo-lite tooling repo, plus worktree, worktree_root, and resolver_root, all absolute paths with resolver_root equal to worktree; repo_slug an owner slash repo slug; branch the lane branch string; starting_head a resolved commit sha; install_check the string clean or drifted; provisioned_at a host-stamped RFC3339 timestamp; and one optional key worktree_repo, an absolute path naming the git repository the worktree belongs to that cleanup proves ownership against and that a legacy self-hosting record may omit so cleanup falls back to control_repo; any other extra key, missing required key, wrong type, or wrong source value fails closed.

The host starts the loop process with working directory set to the provisioned worktree and trusted environment variables OCTO_WORKTREE, OCTO_WORKTREE_ROOT, OCTO_CONTROL_REPO, OCTO_REPO_SLUG, OCTO_STARTING_HEAD, OCTO_LANE, and OCTO_PROVISION_RECORD, and these names and the record schema are the frozen stable seam by which the loop learns its host-provisioned worktree.

The invocation layer builds the loop bound-inputs envelope as parsed JSON carrying every field the loop readiness check requires, and it passes that structured envelope as the loop arguments so a stream name is never handed to a JSON-parsing loop entry as a raw command flag.

Each envelope field has one defined source: from the provisioning environment come the local repo path used for git operations and the worktree, both from OCTO_WORKTREE because the loop runs inside its provisioned worktree, the worktree root from OCTO_WORKTREE_ROOT, the repo slug from OCTO_REPO_SLUG, and the shaping head and PR head from OCTO_STARTING_HEAD; from stream and brief data come issue, PR number, PR base, branch, topology revision, conversation log references and cutoff, and acceptance criteria; from live reads at the shaping head come spec revision, spec blobs, ADR blobs, the resolved worker-role contract hash, the fresh PR head confirmation, and the Linear revision, fingerprint, and state; and from the shaping-review journal come the clear shaping verdict, its head, the reviewer receipt, and the verdict input bindings, so no required field including the local repo path and the contract hash is unsourced and a missing source fails closed before the envelope is passed.

Provisioning verifies the created worktree by reality before emitting the record, requiring its top-level to equal itself, its git-common-dir to belong to the control repo, its origin remote to resolve to the repo slug, its head to equal the resolved starting commit, its branch to equal the lane branch, and its tree to be clean.

Provisioning keeps role, instruction, and skill wiring correct by requiring, as the per-lane guarantee for an octo-lite self-hosting lane whose worktree carries canonical roles.toml and role files, that the role resolver root is the provisioned worktree so core role spawns resolve there rather than through a drift-prone shared symlink, and it records the installed-surface check state as clean or drifted without silently overwriting a foreign-owned link, since the installer refuses to replace a nonmatching existing target and a genuine drift is routed to the installed-surface owner rather than repaired in place, while the worktree own AGENTS.md and CLAUDE.md are correct by construction and verified.

Provisioning separates the git repository the worktree belongs to from the octo-lite tooling repository: the worktree's own repository is the delivery target and can be a target repo that never carries roles.toml or the installer because octo-lite is installed foreground tooling and never a target dependency, while the octo-lite control repository owns role_resolver.py, roles.toml, and the installer and becomes the record's control_repo and thus OCTO_CONTROL_REPO, from which the loop resolves the worker role, so provisioning verifies the canonical registry and runs the installed-surface check against that octo-lite control repository rather than the target worktree, and the two coincide for an octo-lite self-hosting lane.

A target lane is loop-runnable, not merely provisioned, because OCTO_CONTROL_REPO points at the octo-lite control repository so the loop resolves octo-lite tooling there while the provisioned worktree remains the resolver root of record and the working directory for git operations, and the target repository never supplies resolver code; when the octo-lite control repository is omitted it defaults to the worktree's own repository for a self-hosting lane, and a target lane that names no octo-lite control repository with a canonical registry fails closed rather than producing a record that cannot resolve roles.

When host provisioning is in force the loop trusts its working directory and provisioning record as the workspace identity source, and the interim receipt-binding and live-read identity anchors are retired as the identity trust source because provisioning makes them redundant, leaving any residual reality read to the integrating loop as a non-authoritative implementation choice, not a durable requirement.

The integrated delivery loop makes host provisioning the identity trust root: before any Shaped to Todo fire or worker spawn a host-controlled read-only subagent reads the out-of-tree provisioning record named by OCTO_PROVISION_RECORD and resolves the frozen launch environment itself, and the loop rejects unless the record source is host-provisioned-worktree with schema version 1 and resolver root equal to worktree and the envelope repo slug, worktree, starting head, and branch match both the record and the resolved environment, since a child cannot forge its own launch environment and the process working directory equals OCTO_WORKTREE.

The retired receipt-at-OCTO_RECEIPT anchor no longer sources identity trust, and the live git read of the provisioned worktree stays only as a cheap defense-in-depth reality confirm that still rejects an accidentally misrouted worktree on a foreign branch or a foreign origin remote, so its presence never restores the receipt as the trust root and its removal would not weaken the provisioning trust root.

A worker pass binds no durable TOML receipt; its binding proof is the workflow journal plus a schema-forced acknowledgment echo of the exact bound inputs, meaning role, repo, issue, PR, starting HEAD, spec blobs, and contract hash, which the owning orchestrator deterministically verifies before any mutation phase, so a role, repo, issue, or PR substitution fails the echo exactly as a HEAD mismatch does.

A durable TOML receipt binds only a persistent session: spawn, parent and reply route, role, runtime, repo, instructions, control identity, required prior gates, and for an issue orchestrator the exact bound issue identifier.

An issue orchestrator's persistent receipt additionally binds its exact issue identifier under an issue table, written only by the launcher resolve path and never self-written by a lane agent, so the Shaped-transition gate that requires the caller receipt to name that exact issue admits it, while a meta-operator or epic-orchestrator receipt binds no issue and omits the table.

The mutating worker identity is bound to the isolated delivery branch the host gates, and the independent observer verification plus the fresh pre-push readback bind that identity before any host push.

The admission matrix, Linear-state gate, manifest-shape and launch-revision revalidation, and bound-result checks run as deterministic workflow-layer helpers the orchestrator calls before each subagent spawn and before every mutation phase.

The initial launch context and receipt state only execution location, operator loopback reachability, and required review delivery path.

Unless launch context explicitly confirms operator loopback access, every role treats loopback as unreachable and never presents a localhost URL as usable operator access.

Durable access behavior belongs in octo-lite specs and roles, durable target-specific conventions belong in target AGENTS.md, current access facts belong in the receipt or stream status, and temporary routes belong only in stream status.

Role, declared purpose, and launcher-resolved capability set are one validated combination checked before any worktree creation or provider bootstrap, and an invalid combination fails closed leaving no worktree, receipt, provider session, or partial launch state.

A shaping-review purpose is admitted only for role shaping-reviewer, or role orchestrator whose launcher-resolved capability set includes shaping; every other role, and an orchestrator without resolved shaping capability, is rejected.

A delivery purpose is admitted only for role implementer, code-reviewer, qa-capture, qa-reviewer, or the Read-restricted tdd-observer; every other role is rejected, and tdd-observer is admitted only as a Read-restricted subagent.

Every persistent-session bootstrap entrypoint revalidates its receipt against its manifest shape and its launch revision immediately after receipt readback and before the provider runner is invoked, and again before marking the receipt ready or bootstrap-verified; every worker subagent path performs the equivalent revalidation in the workflow layer against the journalled bound inputs before spawn and before any mutation phase, so a shape violation, an altered combination, or a revision mismatch is rejected with no provider process or subagent started.

A receipt already marked bootstrap-verified with a recorded provider session is accepted idempotently on the re-verify path, re-confirming the exact recorded provider session and the same acknowledgment facts under full receipt revalidation, instead of being refused or forced through a fresh full session resolve the provider rejects as a duplicate; a receipt marked bootstrap-verified with a missing or blank recorded provider session is an inconsistent bootstrap state that is rejected fail closed before any provider invocation instead of being misclassified as fresh, and no failure on a receipt claiming prior verification ever removes the previously verified session's worktree.

Only one durable receipt shape remains: the generic persistent launch receipt for role meta-operator or orchestrator carrying no pass purpose, preserving its existing ready semantics; every worker pass, including reconcile, binds through the workflow journal under the role-purpose-capability matrix, and a pass purpose injected onto a persistent receipt, a role substituted for its shape, or an unknown manifest type is rejected.

Shaped-transition authority binds to the verified persistent issue-orchestrator receipt whose launcher-resolved skill set includes shaping, together with a journalled clear shaping verdict carrying workflow-native reviewer provenance of relay-verbatim rollout-record grade at the exact repo, PR, and freshly verified PR HEAD, so an unqualified verdict or any interim authority path is rejected once this lands, plus the machine-checkable operator-intent record that tooling produces at action time under prior explicit operator intent, stamping that same exact head; a worker shaping-review pass produces the verdict but never carries the transition authority itself, and every other role, an orchestrator without resolved shaping capability, a missing or stale journalled verdict, or a missing, fabricated, or wrong-head operator-intent record carries no shaping authority, while no separate operator sign-off is requested per default-to-action law.

Shaped-transition authority is exact-issue authority: the journalled verdict binds the exact issue and the caller session must be that issue's registered issue orchestrator in the live stream registry, so an epic orchestrator, or an issue orchestrator owning any other issue, is rejected even with shaping capability.

A reconcile purpose is admitted only for role reconciler spawned as a Read-restricted workflow subagent; every other role is rejected, and reconcile carries no shaping or delivery authority.

An OpenAI review pass needing live GitHub or Linear reads bootstraps read-only, then resumes with explicit workspace-write sandbox and network access, and launch rejects any read-only review-pass worktree HEAD or status change found after bootstrap or after the resumed pass.

Every OpenAI resume call selects its sandbox through -c sandbox_mode=... config, never the top-level -s flag, because the installed CLI's resume subcommand rejects -s while its top-level exec bootstrap still accepts it.

The OpenAI review stage is a native workflow stage that the workflow owns and codex is the engine, and its relay runs codex exec in the background, detached, to completion rather than as a single foreground timeout-bound call, while the read-only bootstrap stays the top-level -s read-only flag that the codex exec bootstrap accepts and only the resume selects its sandbox through -c sandbox_mode.

On any cut before the codex final assistant message exists, the relay resumes the same session through codex exec resume until that final message is produced, and it never restarts the pass from scratch and never spawns a new session, because a restart loses in-progress rollout and creates provenance ambiguity.

The claimed session id is stable across resumes, so a resumed-to-completion session has the same provenance standing as a single-shot completion, and the same rollout record backs the final message through the unchanged independent rollout reader and relay-verbatim and resume-sandbox gates.

Launch admits a shaping-review purpose only for Linear state Ideas, Todo, Shaped, or In Progress, and admits a delivery purpose only for Linear state Shaped, Todo, or In Progress; every other state, including Awaiting Accept and terminal states, is rejected for both purposes.

Admitting In Progress for both purposes lets shaping re-review run after material reconciliation and lets implementation, code review, re-review, and QA review workers launch while the stream orchestrator holds In Progress, without weakening the exact shaping-verdict checks a delivery purpose still requires.

Loop fire acts on Linear state Shaped and its first act is the mechanical Shaped -> Todo transition; shaped-not-started work therefore rests at Shaped until fire and enters Todo as the first delivery step before any delivery-purpose worker launches against it, and launch never treats Shaped as a direct In Progress predecessor.

Fresh workspace and retry lifecycle

Workspace and retry

One clean control clone supplies shared Git objects to fresh per-pass worktrees.

Targets may reuse native safe download caches while mutable dependency trees and build outputs remain worktree-local.

Target-owned bootstrap fails fast on wrong repo, base, remote, tools, dependencies, environment, database access, disk health, or validation capability.

Before spawn, check disk health, isolated worktree, conflicting ownership of branch, fixture, port, or deployment resource, and active provider overload; unsafe work waits.

Admission that passes then provisions the isolated worktree by creating it from the control repo at the lane branch and starting commit, so admission is the guard and provisioning is the act that yields the exclusive per-lane worktree the loop runs inside.

Provisioning is idempotent for the same lane at the expected repo, branch, and head, reusing an already-provisioned worktree without error, while a path bound to a different lane or branch is refused.

Lane provisioning shares the existing worker-pass create-and-verify path that guards, creates from the control repo, and checks top-level, common-directory, head, branch, and clean tree, and it adds the origin-slug reality check and the same-lane idempotent-reuse disposition, so the existing fresh-absent worker-pass behavior is preserved unchanged while lane provisioning reuses rather than rejects its own already-provisioned worktree.

State-mutating QA verifies starting state, avoids fixtures under concurrent mutation, and records its fixture; read-only QA may share stable fixtures.

Completed worker worktrees are removed only after durable handoff, commit, and push state exists, while dirty worktrees stop for inspection.

Committed work on an isolated branch is not dirty, because a durable commit with an unchanged worktree has zero dirty lines and reset is the wrong tool for it.

A rejected delivery pass leaves its isolated branch unpushed as the correct host-non-push abort, rather than a destructive reset of committed work the host never pushed.

A genuinely dirty or diverged worktree still stops for inspection, so the committed-branch abort reconciles with clean-abort inspection without contradiction.

A pass that fails before its acknowledgment echo is verified removes its worktree on the error path only when that worktree is pristine, meaning zero dirty lines and unchanged HEAD, and only under proven host-provision ownership, either because the failing flow itself provisioned that worktree at a required-absent path or because a host-provision record at its host-known location, carrying source host-provisioned-worktree, schema version 1, and a recorded worktree equal to this exact worktree, proves workspace-admit created and owns it, where a record proves ownership only when the live worktree also matches the record's provision instance, its git common directory resolving to the record's worktree_repo, the git repository the worktree belongs to, or to the record's control_repo when a legacy self-hosting record omits worktree_repo and the two coincide, and its checked-out branch equal to the record's branch; a record-authorized removal retires the proving provision record before the worktree is removed, and a failed retirement halts the removal and preserves the worktree, so the only failure bias is a leaked worktree and a stale record never re-authorizes deleting a later hand-created worktree at the same conventional path, a foreign or hand-created worktree with no such matching record is preserved and never force-removed, an abandoned owned clean worktree still never blocks the next fresh pass at the same path, and a dirty or diverged abort still stops for inspection.

A completed read-only reconcile worktree is removed only after its result is bound and the journal entry is durably persisted, while a failed, diverged, or dirty reconcile worktree is preserved for inspection.

Only failed or interrupted passes add a compact failure record with exact attempt, original error, category, useful work, owner, and next action.

The category is exactly one of input or contract, environment, provider or session, implementation or validation, or external dependency.

Timeout alone is not failure and cannot launch duplicate work.

The owner inspects and preserves useful work before launching a fresh retry instance.

A fleet outage requires the original error, workspace inspection, a fresh exact-model probe, and lane, session, account, provider, or fleet classification.
