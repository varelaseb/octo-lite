Role and model topology

The complete LLM role roster is meta-operator, orchestrator, shaping-reviewer, implementer, code-reviewer, qa-capture, qa-reviewer, and reconciler.

The issue orchestrator performs shaping mode by resolving the issue-shaper skill in its own persistent session, never a second persistent role or tab.

Every fix uses a fresh implementer instance and every re-review uses a fresh code-reviewer instance rather than separate role contracts.

Every nested stream owner uses the same orchestrator contract.

Every issue has one and only one dedicated Opus issue orchestrator.

A multi-issue epic adds a separate Opus epic orchestrator above its issue orchestrators, producing two Opus layers below Fable.

Fresh workers report to their issue orchestrator; issue orchestrators report to the epic orchestrator; the epic orchestrator reports upward to Fable or its declared parent.

A parent flattens relevant effective rules into the child brief; the child acknowledges them only at spawn or after a material brief revision.

Acceptance recording, merge, evidence publication, Linear transition, preproduction promotion, and live traffic shift are deterministic operator-authorized helpers rather than LLM roles.

The control panel displays deterministic gates and helpers separately from LLM roles.

The meta-operator uses exact claude-fable-5 at xhigh effort in auto permission mode.

Fable 5 uses its default 1M context without a redundant [1m] suffix.

An unavailable Fable 5 fails visibly and is never silently replaced by Opus or another model.

Every stream or subordinate orchestrator uses exact claude-opus-4-8[1m] at high effort in auto permission mode.

An unavailable orchestrator model fails visibly and is never silently replaced by another version or family.

Implementation and every fix pass use a fresh exact Anthropic Sonnet 5 instance at xhigh effort.

Code review and every re-review use a fresh GPT-5.6 Sol instance at high effort.

The final post-grill shaping review uses a fresh GPT-5.6 Sol instance at xhigh effort on the fast service tier.

QA capture uses a fresh exact Anthropic Sonnet 5 instance at high effort.

QA review uses a fresh GPT-5.6 Sol instance at high effort on the fast service tier.

Each reconciliation sweep uses a fresh exact claude-sonnet-5 instance at high effort with no fallback.

Deterministic helpers perform exact checks and provide one compact normalized read-only snapshot; Sonnet classifies only changed, missing, stale, contradictory, or needs_fable.

The reconcile gateway reads the exact snapshot file bytes from a regular contained path and hashes them, rejecting a missing, substituted, corrupt, or digest-mismatched snapshot before any worktree or provider launch.

Reconcile PR facts bind normalized PR state, review decision, and status check rollup outcomes alongside head, base, and branch, so a change in any of them changes the snapshot fingerprint.

Normalized status check facts sort on name, current status, and outcome so duplicate-name checks in any API order, or with differing outcomes or in-progress states, never collapse or reorder the same rollup's fingerprint.

The reconciler's canonical spec and ADR sweep requires every declared canonical path to exist as a readable regular file at the exact control HEAD, and a missing, renamed, unreadable, non-file, or empty resulting source set fails the sweep closed before any snapshot or worktree is produced.

Each target repo root AGENTS.md declares its own canonical spec and ADR paths as exact comma-separated Canonical spec paths and Canonical ADR paths lines, and octo-lite's own AGENTS.md declares its own paths the same way rather than a hardcoded constant.

A target initialization template declares exactly one Canonical spec paths placeholder and exactly one Canonical ADR paths signal in root AGENTS.md alongside the spec-format signal.

Shaping replaces the spec placeholder with exact target-owned canonical paths before readiness, and no role, skill, workflow, or octo-lite dependency is copied into the target.

The canonical sweep fails closed on an absent declaration, a missing, non-blob, empty, duplicate, malformed, escaped, or symlinked declared path, or a path declared under both the spec and ADR label.

Canonical ADR paths: none is the sole explicit no-ADR representation and binds an empty ADR blob list rather than a path.

Any other empty value, sentinel, a mix of none and a path, or a repeated Canonical ADR paths declaration fails the sweep closed, and at least one canonical spec path remains mandatory.

Zero or more than one signal line for either label fails the sweep closed, a prefix lookalike line never counts toward that total, and duplicate paths within one otherwise valid line still fail closed.

A sweep refetches and validates current Linear, GitHub PR, target HEAD, and canonical spec and ADR facts before any unchanged decision or snapshot persistence, and the compared fingerprint binds all of them rather than only declared stream facts.

The sweep captures one target HEAD and passes it as the gateway's expected HEAD, and the gateway compares its own freshly read current repo HEAD against that expected HEAD before validating any source or creating a worktree.

Any HEAD change between capture and gateway comparison, including an unrelated commit that leaves canonical blobs unchanged, fails before any provider launch and persists no snapshot, receipt, result, worktree, or sweep state.

The persisted receipt and the child worktree bind the same expected HEAD.

A snapshot, sweep state, and worktree are persisted only after every bound fact validates; a failed target HEAD read or canonical sweep leaves none of them behind.

The sweep never persists snapshot.md under the final sweep directory before Linear, PR, HEAD, spec, and ADR validation all succeed, and a failing gateway leaves no sweep artifact directory or state behind.

A temporary input required only for digest verification lives outside the final sweep directory and is removed on every failure, while a successful sweep still persists its atomic snapshot, receipt, result, and no-op state.

The persisted receipt binds reconcile.snapshot_path to the final persisted snapshot.md, never the temporary digest-verification input, and that exact file exists with matching bytes and digest before bootstrap starts.

Every caught pre-provider failure, including a receipt persistence failure that happens after snapshot persistence, leaves no final sweep snapshot, receipt, or sweep directory and no pending digest-verification input behind.

Reconciler bootstrap opens the exact file at the receipt-bound reconcile.snapshot_path, hashes its bytes, and refuses acknowledgment when that file is missing or its digest does not match reconcile.snapshot_digest.

Sonnet cannot mutate sources, override deterministic mismatches, investigate open-endedly, or silently resolve ambiguity.

Missing or unparseable input and semantic ambiguity return needs_fable; Fable judges the case.

Orchestrated Claude Workflows are the primary loop runtime rather than an adapter target.

Each roles/<role>.md is the sole canonical prose contract for that role.

roles.toml exclusively owns provider, exact model, effort, permission mode, tools, required and conditional skills, and contract path.

Role prose owns responsibilities, authority, stop conditions, inputs, and outputs without repeating machine configuration.

One small resolver combines the canonical role contract and machine mapping before every Workflow agent invocation or Sol relay.

Claude Markdown adapters are generated launcher inputs and diagnostic surfaces, never alternate role-bearing spawn paths.

Codex has no generated custom-agent adapter; Codex custom agents are not part of correctness or fallback for any role.

An OpenAI-backed role pass is one explicit codex exec relay launched by octo-launch, carrying exact model, effort, service tier, worktree, receipt, and the canonical contract as the exec prompt, and it never discovers or depends on a Codex custom agent.

The relay fails closed unless the exact effective provider, model, and effort are independently read back from the Codex session's own rollout record before any mutation.

Claude Workflows load canonical role prose verbatim and Sol relays receive the same prose without copied workflow literals.

The launcher loads every required skill plus only matched conditional skills and records the actual resolved set.

The launch receipt records role path and git blob, roles.toml revision, exact runtime values, and resolved skills.

Adapter generation fails if a target platform cannot express a required tool, mode, or contract field.

roles.toml contains no target rules, issue scope, live state, secrets, or operator decisions.

One path-filtered CI dry run proves role blob, map revision, model, mode, skills, target instruction blob, repo, worktree, and starting HEAD reached the spawned session.

If the Workflow runtime cannot read canonical files directly, a deterministic prebuilt Workflow injects the exact role text and source blob without handwritten prompt literals.

A worker instance is never resumed for another pass and --last is prohibited.

A completed orchestrator stream archives its control record and terminates rather than accepting later work.

Fail-closed launch pipeline

Role launch

octo-launch is the sole mechanism that bootstraps, verifies, and runs a role pass; a Claude Workflow performs deterministic gating over an already-completed, receipt-bound pass result and never spawns a worker session itself.

The launch helper performs final exact Linear and PR and HEAD readback before dispatch.

It provisions a fresh contained worktree from the declared base and starts the exact role, model, effort, mode, and skill set.

It captures the exact provider session ID and requires a compact bootstrap acknowledgment before mutation.

Workspace containment is checked exactly at provisioning and child bootstrap, not before every command.

The receipt binds spawn, parent and reply route, role, runtime, repo, instructions, issue, spec, PR, topology, starting HEAD, and required prior gates.

The initial launch context and receipt state only execution location, operator loopback reachability, and required review delivery path.

Unless launch context explicitly confirms operator loopback access, every role treats loopback as unreachable and never presents a localhost URL as usable operator access.

Durable access behavior belongs in octo-lite specs and roles, durable target-specific conventions belong in target AGENTS.md, current access facts belong in the receipt or stream status, and temporary routes belong only in stream status.

Role, declared purpose, and launcher-resolved capability set are one validated combination checked before any worktree creation or provider bootstrap, and an invalid combination fails closed leaving no worktree, receipt, provider session, or partial launch state.

A shaping-review purpose is admitted only for role shaping-reviewer, or role orchestrator whose launcher-resolved capability set includes shaping; every other role, and an orchestrator without resolved shaping capability, is rejected.

A delivery purpose is admitted only for role implementer, code-reviewer, qa-capture, or qa-reviewer; every other role is rejected.

Every executable launcher entrypoint, whether a standalone bootstrap from an already-persisted receipt, a standalone acknowledgment verification, or a bootstrap composed into a full launch or reconcile pass, revalidates the receipt against its manifest shape and its launch revision at the shared boundary immediately after receipt readback and before the provider runner is invoked, and again before marking any receipt ready or bootstrap-verified, so a manifest-shape violation, a pre-fix or altered invalid combination, or a launch-revision mismatch is rejected with no provider process started and no receipt marked ready or bootstrap-verified.

Manifest-aware revalidation admits an octo-lite-pass receipt only when it satisfies the shaping-review or delivery role-purpose-capability matrix, an octo-lite-reconcile receipt only for role reconciler carrying no pass purpose, and a generic persistent launch receipt only for role meta-operator or orchestrator carrying no pass purpose, each preserving its existing ready semantics; a pass purpose injected onto a non-pass receipt, a role substituted for the receipt's manifest shape, or an unknown manifest type is rejected.

A canonical launcher-produced receipt binds a [shaping] table with repo, PR number, and exact starting HEAD from the freshly verified PR only for role orchestrator whose launcher-resolved capability set includes shaping and purpose shaping-review; every other role, an orchestrator without resolved shaping capability, or any other purpose carries no shaping authority.

An OpenAI review pass needing live GitHub or Linear reads bootstraps read-only, then resumes with explicit workspace-write sandbox and network access, and launch rejects any read-only review-pass worktree HEAD or status change found after bootstrap or after the resumed pass.

Every OpenAI resume call selects its sandbox through -c sandbox_mode=... config, never the top-level -s flag, because the installed CLI's resume subcommand rejects -s while its top-level exec bootstrap still accepts it.

Launch admits a shaping-review purpose only for Linear state Ideas, Todo, Shaped, or In Progress, and admits a delivery purpose only for Linear state Shaped, Todo, or In Progress; every other state, including Awaiting Accept and terminal states, is rejected for both purposes.

Admitting In Progress for both purposes lets shaping re-review run after material reconciliation and lets implementation, code review, re-review, and QA review workers launch while the stream orchestrator holds In Progress, without weakening the exact shaping-verdict checks a delivery purpose still requires.

Loop fire acts on Linear state Shaped and its first act is the mechanical Shaped -> Todo transition; shaped-not-started work therefore rests at Shaped until fire and enters Todo as the first delivery step before any delivery-purpose worker launches against it, and launch never treats Shaped as a direct In Progress predecessor.

Fresh workspace and retry lifecycle

Workspace and retry

One clean control clone supplies shared Git objects to fresh per-pass worktrees.

Targets may reuse native safe download caches while mutable dependency trees and build outputs remain worktree-local.

Target-owned bootstrap fails fast on wrong repo, base, remote, tools, dependencies, environment, database access, disk health, or validation capability.

Before spawn, check disk health, isolated worktree, conflicting ownership of branch, fixture, port, or deployment resource, and active provider overload; unsafe work waits.

State-mutating QA verifies starting state, avoids fixtures under concurrent mutation, and records its fixture; read-only QA may share stable fixtures.

Completed worker worktrees are removed only after durable handoff, commit, and push state exists, while dirty worktrees stop for inspection.

A completed read-only reconcile worktree is removed only after its result is bound and the receipt is durably persisted, while a failed, diverged, or dirty reconcile worktree is preserved for inspection.

Only failed or interrupted passes add a compact failure record with exact attempt, original error, category, useful work, owner, and next action.

The category is exactly one of input or contract, environment, provider or session, implementation or validation, or external dependency.

Timeout alone is not failure and cannot launch duplicate work.

The owner inspects and preserves useful work before launching a fresh retry instance.

A fleet outage requires the original error, workspace inspection, a fresh exact-model probe, and lane, session, account, provider, or fleet classification.
