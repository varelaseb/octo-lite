Dashboard authority boundary

Dashboard boundary

The optional dashboard lives inside octo-lite under dashboard/, with frontend dependencies and build output isolated from the core workflow.

It is not a sibling repository and does not require independent schema-version coordination with octo-lite.

Core installation and workflow execution do not build, launch, or depend on the dashboard unless explicitly invoked.

Hosted mode remains a static read-only projection; local operator mode may edit only approved canonical octo-lite files, validate, and show a git diff.

Local edit paths are exactly roles/*.md, roles.toml, profile/AGENTS.md, and skills/*/SKILL.md.

Generated adapters, workflows, target-repo instructions, specs, and runtime state remain read-only in the dashboard.

A save validates and shows the git diff but never commits or pushes; no per-file edit registry exists.

The dashboard role inspector shows current roles, model, effort, mode, Linear mappings, actual resolved skills, workflow spawn graph, symlink health, editable canonical role prompts, and the generic system or profile prompt.

Validated prompt or mapping changes affect only newly spawned sessions; existing sessions remain pinned to their launch contract.

The panel derives target values from existing repo instructions, specs, git, Linear, PRs, and evidence manifests; a target has no dashboard adapter by default.

Only a shaped, proven non-derivable presentation mapping may add .octo-lite/dashboard.toml; it contains no agent instructions, lifecycle state, secrets, or product dependency.

Acknowledged message lifecycle

Messaging

Every Herdr message uses herdr-say, which attempts transport at a verified safe prompt or durably queues without acting on a modal.

herdr-say and herdr-drain never infer composer state, consumption, or delivery from pane transcript text.

A send or Enter transport attempt leaves a message pending; retry reuses the same message ID, retries transport only, and never re-pastes text that already landed.

A successful queued transport persists the pending durable status before removing its inbox retry item; a persistence failure retains the retry item and the queued status.

Visible, injected, pending, or queued TUI text is not acknowledgment.

Commands, operator rulings, ownership transfers, and blocking questions require explicit recipient acknowledgment before dependent action.

Acknowledgment has no arbitrary timeout because dependency, not duration, determines blocking.

Completion requires the named durable artifact or observable outcome, not acknowledgment alone.

An info-kind message stays best-effort transport and never claims acknowledged delivery.

One-level status compaction

Session-durable reconciliation sweep

Operator supervision

Fable is the sole writer of the reconciled whole-operation view but does not ingest every raw log.

Upward status leads with product outcome, current gate, blocker, and next operator action; tracker and run identifiers are secondary.

An operator gate never presents or requests an exact head or hash; the ask carries plain-language what-changed context and rendered pages, while tooling stamps the exact head at approval time and records the binding on the PR and status surfaces.

Proactive operator messages are limited to required decisions, blockers, failures, and meaningful gate or outcome changes.

A target operations view becomes primary only when it is accurate and digestible across all workstreams; until then Fable supplies the complete operation update.

Each stream uses one parent-owned brief.md and one child-owned status.md in the operator control directory.

Status compacts upward one parent at a time, and leaf workers do not all message Fable.

One session-durable host timer invokes operator-say sweep throughout the operator lifecycle, including idle periods.

A wake runs deterministic exact checks and may spawn one fresh Sonnet reconciler over the compact normalized snapshot.

A wake with no delta is a cheap no-op and does not remove the timer.

The human may converse directly with an orchestrator, and stream-local rulings apply after durable status capture while cross-stream effects wait for Fable reconciliation.

Atomic Fable handoff

Operator handoff

Only an explicit operator request starts a Fable-to-Fable handoff.

The outgoing Fable writes one concise immutable Markdown handoff at handoffs/<zero-padded-revision>.md containing objective, stream tree, exact facts, pending decisions, authorities, blockers, control path, timer, and reply routes.

The fresh successor loads canonical instructions, reconciles the handoff against source systems, and declares readiness before cutover.

One operator-owner.toml stores schema version, exact owner session ID, handoff revision, and control-directory identity.

A locked expected-owner and revision check plus atomic TOML rename is the sole authority commit.

The timer and top-level child notifications resolve the owner through operator-say, so the next message observes the same atomic commit.

The old operator becomes read-only at commit and may only forward late direct messages until cleanup.

A dead or ambiguous owner record has no agent-callable recovery path; only an explicit manual operator takeover outside role authority may replace it.

No probe, self-asserted authorization, timeout, or automatic failover transfers authority; a blocked owner record stays blocked until manual takeover.

The control-directory streams/ tree is the sole sweep-visible stream registry; Fable is the sole writer of the registry itself, meaning which streams exist, while each child writes only its own status.md within its stream.

A receipt streams_json field is a launch-time snapshot only; a reconcile pass reads the live control-directory streams/ registry, never the receipt snapshot field, so a stream added or removed after launch is seen.

Only the parent writes a child brief.

Only the child writes its status.

A status.md is rewritten whole to reflect current truth, never patched piecemeal or appended, so no stale fragment survives a status update.

Only the owner-transfer helper replaces operator-owner.toml.

Only deterministic workflow and gate helpers append trace events.

Operator-owned mutation helpers verify the exact caller against operator-owner.toml; ordinary read-only commands do not.

Persistent session label grammar

Herdr session labels

The sole Fable tab is exactly 🧠 operator, with no issue, PR, or outcome suffix.

🎤 temporarily prefixes an Opus tab while the operator must remain engaged in a direct investigation, grill, diagnosis, or decision conversation.

Ordinary reporting, commands, acknowledgments, handoffs, and parent-child updates do not receive 🎤.

Fable judges the conversation mode from bounded pane context and message provenance, without keyword, turn-count, focus, or timeout rules.

Fable must remove 🎤 as soon as the Opus has enough direction to begin autonomous work and the operator can look away; this is required behavior, not optional cleanup.

◆ immediately precedes an epic identity; normal issue identities are unmarked.

Labels order the optional operator signal, compact issue and PR numbers, then the very short human-readable outcome.

The presentation label omits the tracker team prefix; durable artifacts and receipts retain the full issue key.

The identity token is issue/pr with bare numbers, or only issue before a PR exists.

Only persistent Fable and Opus orchestrator sessions receive Herdr tab labels; workflow subagents run inside the owning Opus session and never create tabs.

Normal unattended tabs use no status emoji, and every label uses the fewest characters that remain unique and human-readable.

The passed persistent spawn role and canonical cwd must exactly equal the receipt's role.name and workspace.worktree before any bootstrap call or pane creation.

Every 🎤 or 🧠 persistent session must have Claude remote control enabled.

Remote control is enabled durably at launch through the launcher mechanism, the claude --rc flag or a remoteControlAtStartup settings injection, never post-hoc.

herdr-spawn treats a 🎤 or 🧠 spawn without remote control as a launch-gate failure and fails closed.

Flipping 🎤 on is a Fable duty that requires ensuring remote control is active on that session.

Lightweight trace model

Lightweight tracing

Tracing aligns to Langfuse session, trace, and observation concepts at the data-contract level only.

No Langfuse deployment, SDK, collector, exporter, service, or trace database is installed.

Each bounded stream may have one append-only trace.jsonl in its existing control directory.

A session is one bounded orchestrator stream or release train.

A trace is one issue-delivery execution.

An observation is one meaningful workflow boundary such as bootstrap, pass completion, verdict, retry, transition, accepted control message, merge, or deployment fact.

Only deterministic workflow and named gate helpers write traces, and language-model agents never write them directly.

Records contain portable IDs, timing, status, compact role and issue attributes, and artifact references but no raw prompts, responses, logs, transcripts, secrets, evidence bodies, or product data.

A trace inherits its stream control-directory lifecycle and has no separate archive, retention timer, or pruning system.

Runtime append failure writes one trace degraded status warning and continues without retrying or blocking gates.

One path-filtered CI checker validates generated happy, failure and retry, and invalid traces without mutating external systems.

Trace conformance does not replace spec-driven TDD or target-repo tests.

Edge cases

A modal or question dialog causes herdr-say to queue rather than inject or submit.

A malformed trace blocks the relevant CI contract check but does not retroactively invalidate authoritative gate artifacts.

A missing or ambiguous owner record blocks every operator-owned mutation until manual operator takeover outside role authority; no role or command may self-authorize recovery.
