Dashboard authority boundary

Dashboard boundary

The optional dashboard lives inside octo-lite under dashboard/, with frontend dependencies and build output isolated from the core workflow.

It is not a sibling repository and does not require independent schema-version coordination with octo-lite.

Core installation and workflow execution do not build, launch, or depend on the dashboard unless explicitly invoked.

Hosted mode remains a static read-only projection; local operator mode may edit only approved canonical octo-lite files, validate, and show a git diff.

Local edit paths are exactly roles/*.md, roles.toml, profile/AGENTS.md, and skills/*/SKILL.md.

Generated adapters, workflows, target-repo instructions, specs, and runtime state remain read-only in the dashboard.

A save validates and shows the git diff but never commits or pushes; no per-file edit registry exists.

The dashboard role inspector shows current roles, model, effort, mode, Linear mappings, actual resolved skills, workflow spawn graph, symlink health, editable canonical role prompts, and the generic system or profile prompt.

Validated prompt or mapping changes affect only newly spawned sessions; existing sessions remain pinned to their launch contract.

The panel derives target values from existing repo instructions, specs, git, Linear, PRs, and evidence manifests; a target has no dashboard adapter by default.

Only a shaped, proven non-derivable presentation mapping may add .octo-lite/dashboard.toml; it contains no agent instructions, lifecycle state, secrets, or product dependency.

Messaging

Every Herdr message uses herdr-say, which attempts transport at a verified safe prompt or durably queues without acting on a modal.

Transport is the herdr 0.7.5 atomic agent prompt, which pastes and submits the message in one server-owned op, replacing the retired 0.7.1 agent send plus paste, settle, and bare-Enter dance.

Because the atomic prompt preserves bracketed paste literally, a message body carries arbitrary multi-line and special-character text verbatim, and the earlier single-line, no-single-quote workaround is retired.

The atomic prompt is still modal-safe: a detected trust or safety dialog defers to a durable queue rather than submitting, and a reported stall, refusal, or dialog defers to pending, so an open modal is never bulldozed.

herdr-say and herdr-drain never infer composer state, consumption, or delivery from pane transcript text.

Transport confirmation is observed state, never the prompt exit code (a prompt can return zero without submitting when the target composer holds residual text): every fire passes --wait --timeout with OCTO_PROMPT_CONFIRM_TIMEOUT_MS milliseconds (positive integer, default 15000, invalid values degrade to the default with a warning), only a matched post-submission agent state (idle, done, or blocked) confirms, and agent_prompt_stalled, a timeout, a nonzero exit, or any unmatched outcome leaves the message unconfirmed, pending, and retryable with its inbox item under the attempt cap.

An atomic prompt transport attempt leaves a message pending; retry reuses the same message ID and re-fires the same atomic prompt, which never double-submits partially-pasted text, and the retry count is bounded by the per-epoch attempt cap.

Every state transition persists before its inbox item is removed (state-transition precedes item removal), and a queued first fire persists queued to pending before the attempts-increment and before any transport; a persistence failure retains the retry item and the prior status.

Visible, injected, pending, or queued TUI text is not acknowledgment.

Commands, operator rulings, ownership transfers, and blocking questions require explicit recipient acknowledgment before dependent action.

Acknowledgment has no arbitrary timeout because dependency, not duration, determines blocking.

Completion requires the named durable artifact or observable outcome, not acknowledgment alone.

An info-kind message stays best-effort transport and never claims acknowledged delivery; an observed-state-confirmed info submit transitions the message to completed under the lock, and the transported body carries the message id with no ack instruction.

Every status read-decide-write, attempts-increment, and check-then-fire runs under the permanent per-message flock at locks/<id>.lock, including herdr-say's own first transport sequence after INIT; only the INIT exclusive creation itself is exempt, no helper ever holds two locks, nothing ever unlinks a lock, and reclamation is runbook-only.

Message state is persisted atomically before any transport, with the delivery path resolved provisional-then-promote: modal-open publishes queued (modal-queued) with an item, an unresolved operator ask persists pending (unresolved-ask) with no transport, every transport-attempting send persists pending (deferred) and its item first, and only an observed-state-confirmed transport promotes the path to direct.

Transport is a bounded duplicate-prone transport with no delivery guarantee: a message may be delivered zero times (a crash can burn an attempt before the fire, and a cap stall can leave no fire), once, or up to the attempt cap per epoch, so no surface claims a delivery guarantee and the sweep's stalled and undelivered surfacing is the no-delivery detection path; every transported body carries the [msg:<id>] suffix so every duplicate is id-correlated.

The retry cap OCTO_TRANSPORT_ATTEMPT_CAP (positive integer, default 3, invalid values degrade to the default with a warning) bounds fires per epoch from above: a fire first increments and persists transport_attempts under the lock, a message at the cap transitions pending to stalled with no fire, and only the manual herdr-drain --resume starts a new epoch with the counter reset to zero.

One-level status compaction

Session-durable reconciliation sweep

Operator supervision

Fable is the sole writer of the reconciled whole-operation view but does not ingest every raw log.

Upward status leads with product outcome, current gate, blocker, and next operator action; tracker and run identifiers are secondary.

An operator notification never presents or requests an exact head or hash; it carries plain-language what-changed context and rendered pages, while tooling stamps the exact head at action time and records the binding on the PR and status surfaces.

Default to action: prior explicit operator intent is authorization, no action waits on double confirmation, and no blocking human gate exists beyond the lawful stops below.

A former approval gate is an act-then-notify step with a prepared rollback; the agent acts, notifies the operator with plain-language context, and the operator vetoes by rollback.

An operator freeze halts exactly the named loop, such as autonomous review re-dispatch, and nothing else; defect fixes, repairs, and every other act-then-notify duty continue under standing operator intent, and a frozen agent never asks permission to fix a defect.

The only lawful stops are operator-held access the agent does not possess, legally binding irreversible actions without a prepared rollback, and the three operator carve-outs: actual acceptance of delivered work awaiting acceptance, preproduction promotion, and live traffic shifts, all human-gated.

The three carve-outs are gated on the human instruction, not the human's manual action: the human decides and instructs the acceptance, the preproduction promotion, or the live traffic shift, and the operator agent then executes that instruction by recording the acceptance artifact, running the preproduction promotion, or running the traffic shift, never handing the mechanical action back to the human.

The preserved guard is that no agent infers, self-authorizes, or decides acceptance, a preproduction promotion, or a traffic shift on its own, and the operator agent acts only on an explicit human instruction.

Proactive operator messages are limited to required decisions, blockers, failures, and meaningful gate or outcome changes.

A target operations view becomes primary only when it is accurate and digestible across all workstreams; until then Fable supplies the complete operation update.

Each stream uses one parent-owned brief.md and one child-owned status.md in the operator control directory.

Status compacts upward one parent at a time, and leaf workers do not all message Fable.

One session-durable host timer runs the deterministic sweep directly throughout the operator lifecycle, including idle periods, primed to fire without a manual first start; only a changed sweep messages the operator, so no periodic wake message ever enters the operator context.

A wake runs deterministic exact checks and may spawn one fresh Sonnet reconciler over the compact normalized snapshot.

The sweep reports per stream, from real surfaces only (receipt-derived session transcript mtimes, status gate lines, the Herdr inbox and message state, the herdr agent probe, and the status.md anchored Outcome: closed marker): waiting-on-whom with the verbatim open ask, in-flight workflow ids, the undelivered message queue, and an advisory classification.

Undelivered derivation admits only id-shaped fully-published inbox entries, never lock files, dot-temporaries, or foreign files; an item whose state is non-retryable (any terminal, stalled, or an ACK-WAIT path such as promoted-direct residue) is not undelivered and is noted once as residue awaiting drain's uniform removal; a missing or malformed state is noted undelivered-unknown without crashing; and itemless retryable or legacy no-path states stay surfaced while ACK-WAIT states awaiting acknowledgment do not.

The herdr agent probe input is the grammar-stable non-blocking agent get of the delivery-lifecycle sweep-dead-agent law, bounded per stream, and a probe failure degrades gracefully to the time-based classification with a noted error per sweep-watchdog-degrade.

Classification precedence is waiting-on-operator over closed over dead over waiting over suspected-stuck over active, with closed and dead defined by the delivery-lifecycle sweep-closed-stream and sweep-dead-agent laws.

Activity means session-transcript activity only and status-file age never counts; an idle stream without a recorded wait-owner is suspected-stuck only when it is neither closed nor dead, and an ask older than later session activity is presumed consumed.

The liveness report lives in the sweep output and one control-level file, never inside the digested reconcile snapshot, so digests verify byte-exactly and active streams never churn the fingerprint.

A wake with no delta is a cheap no-op and does not remove the timer.

The human may converse directly with an orchestrator, and stream-local rulings apply after durable status capture while cross-stream effects wait for Fable reconciliation.

Atomic Fable handoff

Operator handoff

Only an explicit operator request starts a Fable-to-Fable handoff.

The outgoing Fable writes one concise immutable Markdown handoff at handoffs/<zero-padded-revision>.md containing objective, stream tree, exact facts, pending decisions, authorities, blockers, control path, timer, and reply routes.

The fresh successor loads canonical instructions, reconciles the handoff against source systems, and declares readiness before cutover.

One operator-owner.toml stores schema version, exact owner session ID, handoff revision, and control-directory identity.

A locked expected-owner and revision check plus atomic TOML rename is the sole authority commit.

The timer and top-level child notifications resolve the owner through operator-say, so the next message observes the same atomic commit.

The old operator becomes read-only at commit and may only forward late direct messages until cleanup.

A dead or ambiguous owner record has no agent-callable recovery path; only an explicit manual operator takeover outside role authority may replace it.

No probe, self-asserted authorization, timeout, or automatic failover transfers authority; a blocked owner record stays blocked until manual takeover.

The initial-launch launcher creates the first owner record at revision zero and refuses when an owner record already exists.

The distinct live-handoff launcher requires an existing owner record, spawns one successor that coexists with the current owner under the same invariant control directory, writes the successor session artifacts under its own operator directory, and writes no owner record itself.

The live-handoff launcher reads the current owner record and the next-revision handoff artifact, then pre-derives and prints the exact successor-ready command for the successor and the exact owner-transfer command for the current owner to run after the successor declares readiness.

The launcher handles only the live path, cannot and does not distinguish a live from a dead owner, and never probes liveness; it prints exactly one runnable owner-transfer command bound to the current owner identity, which is the normal owner-run handoff-atomic step and not a new recovery command, and it never derives a second command from any other or prior identity and never adds an authorization flag.

The owner-transfer caller-identity gate is a pre-existing octo-control property that the launcher neither strengthens nor weakens, the launcher introduces no new authentication, and any future session-authentication hardening is a separate operator-initiated change.

The launcher output carries a one-line boundary notice that the printed command is to be run only as the live current owner and that a dead or ambiguous owner is recovered solely by the manual operator takeover outside role authority on operator instruction, and the skill documents this same boundary and the pre-existing caller-identity property explicitly.

Persistent session label grammar

Herdr session labels

The sole Fable tab is exactly 🧠 operator, with no issue, PR, or outcome suffix.

🎤 temporarily prefixes an Opus tab while the operator must remain engaged in a direct investigation, grill, diagnosis, or decision conversation.

Ordinary reporting, commands, acknowledgments, handoffs, and parent-child updates do not receive 🎤.

Fable judges the conversation mode from bounded pane context and message provenance, without keyword, turn-count, focus, or timeout rules.

Fable must remove 🎤 as soon as the Opus has enough direction to begin autonomous work and the operator can look away; this is required behavior, not optional cleanup.

◆ immediately precedes an epic identity; normal issue identities are unmarked.

Labels order the optional operator signal, compact issue and PR numbers, then the very short human-readable outcome.

The presentation label omits the tracker team prefix; durable artifacts and receipts retain the full issue key.

The identity token is issue/pr with bare numbers, or only issue before a PR exists.

Only persistent Fable and Opus orchestrator sessions receive Herdr tab labels; workflow subagents run inside the owning Opus session and never create tabs.

Normal unattended tabs use no status emoji, and every label uses the fewest characters that remain unique and human-readable.

The passed persistent spawn role and canonical cwd must exactly equal the receipt's role.name and workspace.worktree before any bootstrap call or pane creation.

Every 🎤 or 🧠 persistent session must have Claude remote control enabled.

Remote control is enabled durably at launch through the launcher mechanism, the claude --rc flag or a remoteControlAtStartup settings injection, never post-hoc.

herdr-spawn treats a 🎤 or 🧠 spawn without remote control as a launch-gate failure and fails closed.

Flipping 🎤 on is a Fable duty that requires ensuring remote control is active on that session.

Lightweight trace model

Lightweight tracing

Tracing aligns to Langfuse session, trace, and observation concepts at the data-contract level only.

No Langfuse deployment, SDK, collector, exporter, service, or trace database is installed.

Each bounded stream may have one append-only trace.jsonl in its existing control directory.

A session is one bounded orchestrator stream or release train.

A trace is one issue-delivery execution.

An observation is one meaningful workflow boundary such as bootstrap, pass completion, verdict, retry, transition, accepted control message, merge, or deployment fact.

Only deterministic workflow and named gate helpers write traces, and language-model agents never write them directly.

Records contain portable IDs, timing, status, compact role and issue attributes, and artifact references but no raw prompts, responses, logs, transcripts, secrets, evidence bodies, or product data.

A trace inherits its stream control-directory lifecycle and has no separate archive, retention timer, or pruning system.

Runtime append failure writes one trace degraded status warning and continues without retrying or blocking gates.

One path-filtered CI checker validates generated happy, failure and retry, and invalid traces without mutating external systems.

Trace conformance does not replace spec-driven TDD or target-repo tests.

Edge cases

A modal or question dialog causes herdr-say to queue rather than inject or submit.

A malformed trace blocks the relevant CI contract check but does not retroactively invalidate authoritative gate artifacts.

A missing or ambiguous owner record blocks every operator-owned mutation until manual operator takeover outside role authority; no role or command may self-authorize recovery.
